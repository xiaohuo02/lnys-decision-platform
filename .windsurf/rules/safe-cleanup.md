# 安全清理铁律 (任何破坏性文件操作前必读)

## 删除 / 移动 / 重命名任何文件前

- **必须先** `grep_search` 该文件名 (不带扩展名) 在整个项目的引用, 范围至少含 `docs/`, `.windsurf/`, `README*`, `scripts/`, `backend/`, `frontend/`
- 任何文档出现该文件名 → 视为"被引用, 不能动", 至少把引用清单发给用户看
- 操作前必须列完整路径清单给用户, 等明确批准, **不要**先做了再说

## 名字不等于意图 (重要陷阱)

- `_tmp_` / `_archive_` / `_old_` / `_bak_` 前缀**不一定**是垃圾, 可能是:
  - 显式分组的工作区 / 实验配套区
  - 作者主动标注"保留"的归档
  - 文档引用的工具集 / 证据库 / 测试 fixtures
- 看到这类前缀, **先读该目录或附近的 README / 索引文档**判断作者意图
- 作者文档里出现 "保留" / "归档" / "工具" / "tools" / "fixtures" / "evidence" / "证据" / "实验" 等关键词 = **明确保留**, 立刻停手

## 执行规范

- 用 `[Microsoft.VisualBasic.FileIO.FileSystem]::DeleteFile(..., SendToRecycleBin)` 送 Windows 回收站
- **禁用** `Remove-Item -Recurse -Force` (除非用户明确指定且确认过引用)
- 删完必须输出: 每个文件大小 / 是否在 git / 还原命令 (`git restore <path>` 或回收站还原步骤)
- 移动 / 重命名目录前列出**所有**引用该路径的位置, 让用户决定是否一并改引用; 不能只改路径不改引用

## 历史教训 (具体反例)

把 `docs/_tmp_kb/` 看成临时垃圾, 删了 8 个 `.sh`. 但 `知识库升级方案.md` 第 1910 行明确写过 "全部保留在 `docs/_tmp_kb/`", 多处文档引用这些脚本作为 KB v2 工具包. 事后 `git restore` 还原. 根因: 仅凭 `_tmp_` 前缀就判定"垃圾", 没先 grep 引用. **下次绝不能再犯**.
