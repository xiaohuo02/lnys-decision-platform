# 全局异常处理修改方案

> 生成日期：2026-04-05
> 基于：`docs/异常处理现状分析.md` 现状分析

---

## 一、设计目标

1. **统一响应格式**：所有 HTTP 响应（成功/失败/异常）共享 `{code, data, message, meta}` 结构
2. **异常分层分类**：业务异常、参数异常、鉴权异常、资源不存在、数据库异常、系统异常各有明确类型
3. **错误码体系**：可追踪、可搜索、前端可映射
4. **安全脱敏**：生产环境不泄露堆栈/SQL/内部路径
5. **日志可观测**：异常日志含 trace_id、用户、接口、异常链
6. **最小改动原则**：沿用现有 `AppError` + `ok()/degraded()` 体系，增量升级

## 二、统一响应结构

### 2.1 成功响应（不变）

```json
{
  "code": 200,
  "data": { ... },
  "message": "ok",
  "meta": {
    "degraded": false,
    "source": "db",
    "trace_id": "abc-123",
    "warnings": []
  }
}
```

### 2.2 错误响应（新增 `meta` 字段 + `error_code`）

```json
{
  "code": 404,
  "data": null,
  "message": "审核单不存在或已处理",
  "meta": {
    "degraded": false,
    "source": "error",
    "trace_id": "abc-123",
    "warnings": [],
    "error_code": "RESOURCE_NOT_FOUND",
    "detail": null
  }
}
```

- `code`：HTTP 语义状态码（200/400/401/403/404/409/422/429/500/503）
- `message`：用户可读消息（中文），**不含内部异常信息**
- `meta.error_code`：机器可读错误码（枚举字符串），前端据此做差异化处理
- `meta.detail`：仅 `DEV/DEBUG` 环境下填充异常详情，生产环境为 null

## 三、异常类型分层

### 3.1 异常类继承树

```
Exception
 └── AppError (base, 保留现有)
      ├── BusinessError          # 业务规则校验失败 (code=400)
      ├── AuthenticationError    # 认证失败 (code=401)
      ├── AuthorizationError     # 权限不足 (code=403)
      ├── ResourceNotFoundError  # 资源不存在 (code=404)
      ├── ConflictError          # 资源冲突/幂等重复 (code=409)
      ├── RateLimitError         # 限流 (code=429)
      ├── ServiceUnavailableError # 服务暂不可用 (code=503)
      └── DatabaseError          # 数据库异常 (code=500, 已定义未使用)
```

### 3.2 各类型定义

```python
# backend/core/exceptions.py 扩展

class AppError(Exception):
    """基类，保持向后兼容"""
    def __init__(self, code: int = 500, message: str = "服务器内部错误",
                 error_code: str = "INTERNAL_ERROR", data=None):
        self.code = code
        self.message = message
        self.error_code = error_code
        self.data = data

class BusinessError(AppError):
    """业务规则校验失败"""
    def __init__(self, message: str, error_code: str = "BUSINESS_ERROR", data=None):
        super().__init__(400, message, error_code, data)

class AuthenticationError(AppError):
    """认证失败（未登录/Token 过期）"""
    def __init__(self, message: str = "认证失败", error_code: str = "AUTH_FAILED"):
        super().__init__(401, message, error_code)

class AuthorizationError(AppError):
    """权限不足"""
    def __init__(self, message: str = "权限不足", error_code: str = "FORBIDDEN"):
        super().__init__(403, message, error_code)

class ResourceNotFoundError(AppError):
    """资源不存在"""
    def __init__(self, message: str = "资源不存在", error_code: str = "RESOURCE_NOT_FOUND"):
        super().__init__(404, message, error_code)

class ConflictError(AppError):
    """资源冲突（重复/幂等）"""
    def __init__(self, message: str = "操作冲突", error_code: str = "CONFLICT"):
        super().__init__(409, message, error_code)

class RateLimitError(AppError):
    """限流"""
    def __init__(self, message: str = "请求过于频繁", retry_after: int = 60):
        super().__init__(429, message, "RATE_LIMITED")
        self.retry_after = retry_after

class ServiceUnavailableError(AppError):
    """服务暂不可用（Agent/模型/外部依赖）"""
    def __init__(self, message: str = "服务暂不可用", error_code: str = "SERVICE_UNAVAILABLE"):
        super().__init__(503, message, error_code)
```

## 四、错误码枚举表

| error_code | HTTP | 含义 | 触发场景 |
|-----------|------|------|---------|
| `BUSINESS_ERROR` | 400 | 业务规则校验失败 | 实验已完成不可重跑、审核单状态不允许操作 |
| `VALIDATION_ERROR` | 422 | 请求参数校验失败 | Pydantic 校验不通过 |
| `AUTH_FAILED` | 401 | 认证失败 | Token 缺失/过期/无效、密码错误 |
| `ACCOUNT_DISABLED` | 403 | 账号已禁用 | 用户 is_active=false |
| `FORBIDDEN` | 403 | 权限不足 | 角色不匹配 |
| `RESOURCE_NOT_FOUND` | 404 | 资源不存在 | run_id/case_id/prompt_id 等查无结果 |
| `CONFLICT` | 409 | 资源冲突 | 名称重复（IntegrityError）、幂等键已执行 |
| `RATE_LIMITED` | 429 | 请求过于频繁 | 滑动窗口限流触发 |
| `INPUT_GUARD_REJECTED` | 422 | 输入安全检查未通过 | analyze 入口 guard 拦截 |
| `SERVICE_UNAVAILABLE` | 503 | 服务/Agent 暂不可用 | 模型文件缺失、ML 流水线未运行 |
| `DB_ERROR` | 500 | 数据库操作失败 | SQL 执行异常（脱敏后返回） |
| `INTERNAL_ERROR` | 500 | 系统内部错误 | 未预期异常（兜底） |

## 五、全局 Handler 注册方案

### 5.1 修改 `core/exceptions.py`

```python
from fastapi import Request, HTTPException
from fastapi.responses import JSONResponse
from fastapi.exceptions import RequestValidationError
from starlette.status import HTTP_500_INTERNAL_SERVER_ERROR
from backend.config import settings

def _error_response(
    code: int,
    message: str,
    error_code: str = "INTERNAL_ERROR",
    data=None,
    trace_id: str | None = None,
    detail: str | None = None,
    headers: dict | None = None,
) -> JSONResponse:
    """统一错误响应构造"""
    body = {
        "code": code,
        "data": data,
        "message": message,
        "meta": {
            "degraded": False,
            "source": "error",
            "trace_id": trace_id,
            "warnings": [],
            "error_code": error_code,
            # 仅开发环境暴露 detail
            "detail": detail if settings.DEV_BACKDOOR_ENABLED else None,
        },
    }
    return JSONResponse(status_code=code, content=body, headers=headers)


async def app_error_handler(request: Request, exc: AppError) -> JSONResponse:
    trace_id = getattr(request.state, "trace_id", None)
    if exc.code >= 500:
        logger.error(f"[AppError] {exc.code} {exc.error_code}: {exc.message} trace={trace_id}")
    else:
        logger.warning(f"[AppError] {exc.code} {exc.error_code}: {exc.message} trace={trace_id}")
    headers = None
    if isinstance(exc, RateLimitError):
        headers = {"Retry-After": str(exc.retry_after)}
    return _error_response(
        exc.code, exc.message, exc.error_code,
        data=exc.data, trace_id=trace_id, headers=headers,
    )


async def http_exception_handler(request: Request, exc: HTTPException) -> JSONResponse:
    """统一处理 FastAPI HTTPException（含中间件抛出的 401/403/429）"""
    trace_id = getattr(request.state, "trace_id", None)
    # 映射 HTTPException.detail → 统一格式
    _CODE_MAP = {
        401: "AUTH_FAILED",
        403: "FORBIDDEN",
        404: "RESOURCE_NOT_FOUND",
        429: "RATE_LIMITED",
    }
    error_code = _CODE_MAP.get(exc.status_code, "HTTP_ERROR")
    message = exc.detail if isinstance(exc.detail, str) else "请求错误"
    logger.warning(f"[HTTPException] {exc.status_code} {error_code}: {message} trace={trace_id}")
    return _error_response(
        exc.status_code, message, error_code,
        trace_id=trace_id,
        headers=dict(exc.headers) if exc.headers else None,
    )


async def validation_error_handler(request: Request, exc: RequestValidationError) -> JSONResponse:
    trace_id = getattr(request.state, "trace_id", None)
    errors = exc.errors()
    # 提取第一个错误作为 message
    first = errors[0] if errors else {}
    field = " → ".join(str(l) for l in first.get("loc", []))
    msg = f"参数校验失败: {field} {first.get('msg', '')}" if field else "请求参数校验失败"
    logger.warning(f"[ValidationError] {msg} trace={trace_id}")
    return _error_response(
        422, msg, "VALIDATION_ERROR",
        data=errors, trace_id=trace_id,
    )


async def unhandled_exception_handler(request: Request, exc: Exception) -> JSONResponse:
    """兜底：捕获所有未处理异常，防止泄露堆栈"""
    trace_id = getattr(request.state, "trace_id", None)
    logger.exception(f"[UnhandledException] {type(exc).__name__}: {exc} trace={trace_id}")
    return _error_response(
        500,
        "服务器内部错误，请稍后重试",
        "INTERNAL_ERROR",
        trace_id=trace_id,
        detail=f"{type(exc).__name__}: {exc}",
    )
```

### 5.2 修改 `main.py` 注册

```python
from fastapi import HTTPException
from fastapi.exceptions import RequestValidationError
from backend.core.exceptions import (
    AppError,
    app_error_handler,
    http_exception_handler,
    validation_error_handler,
    unhandled_exception_handler,
)

app.add_exception_handler(AppError, app_error_handler)
app.add_exception_handler(HTTPException, http_exception_handler)
app.add_exception_handler(RequestValidationError, validation_error_handler)
app.add_exception_handler(Exception, unhandled_exception_handler)
```

**注册顺序要点**：AppError 是 Exception 子类，FastAPI 按注册的具体类型匹配，`AppError` handler 优先于 `Exception` handler。

## 六、各层改造要点

### 6.1 中间件 / 鉴权层

**改前**（`middleware/auth.py`）:
```python
raise HTTPException(status_code=401, detail="Token 无效或已过期")
```

**改后**（两种方案均可）:

- **方案 A（最小改动）**：保持 `HTTPException`，由新增的 `http_exception_handler` 统一转换格式。**推荐此方案**，因为中间件层不宜依赖业务异常类。
- **方案 B**：改为 `raise AuthenticationError("Token 无效或已过期")`。

### 6.2 路由层

| 改前 | 改后 | 说明 |
|------|------|------|
| `raise AppError(404, "xxx不存在")` | `raise ResourceNotFoundError("xxx不存在")` | 语义更清晰，可选改动 |
| `raise AppError(409, "名称重复")` | `raise ConflictError("名称重复")` | 同上 |
| `raise AppError(400, str(e))` | `raise BusinessError(str(e))` | 同上 |
| `raise AppError(503, "xxx暂未就绪")` | `raise ServiceUnavailableError("xxx暂未就绪")` | 同上 |
| `raise HTTPException(404, ...)` in workflow.py | `raise ResourceNotFoundError(...)` | 消除 HTTPException/AppError 混用 |
| `raise HTTPException(422, ...)` in analyze.py | `raise BusinessError(..., error_code="INPUT_GUARD_REJECTED")` | 同上 |

> **注意**：`raise AppError(404, "...")` 的旧写法仍然兼容，因为 `AppError` 基类保留了 `(code, message)` 构造函数。可以逐步迁移，不需要一次性全改。

### 6.3 Service 层

**无需大改**。Service 层的三级降级链和 Redis 静默失败模式设计合理。

**需要改的点**：
1. `fraud_service.py:179` — `AppError(500, f"待审核列表查询失败: {e}")` → 脱敏
   ```python
   raise AppError(500, "待审核列表查询失败")  # str(e) 只写日志
   ```

2. `ops_copilot.py:50` — `str(e)` 泄露到 answer/error 字段
   ```python
   "answer": "运维助手暂时不可用，请稍后重试",
   "error": None,  # 不暴露内部异常
   ```

3. `admin/dashboard.py:85` — `reason=f"dashboard query failed: {e}"`
   ```python
   reason="dashboard query failed"  # 移除 {e}
   ```

### 6.4 Repo 层

**建议渐进改造**：对关键路径（如 `fraud_repo.insert`）区分"无数据"和"查询出错"：

```python
class RepoResult(Generic[T]):
    """Repo 层返回包装，区分成功/失败"""
    ok: bool
    data: T | None = None
    error: str | None = None
```

或更简单的做法：**在 Repo 层只 catch 并 rollback，但 re-raise 为 `DatabaseError`**，让 Service 层决定是否降级：

```python
def save(self, module: str, result: dict) -> bool:
    try:
        ...
        self.db.commit()
        return True
    except Exception as e:
        self.db.rollback()
        logger.warning(f"[repo] save({module}) failed: {e}")
        return False  # 非关键路径保持静默
```

> 当前 Repo 层的静默模式对于"非关键写入"（如缓存回写、审计日志）是合理的。对于"关键业务写入"（如 HITL 案例创建），应让异常冒泡。

### 6.5 Governance 层

`hitl.py` 的 `approve_case` / `edit_case` / `reject_case` 应增加事务保护：

```python
def approve_case(db, case_id, decision_by, note=None):
    case = _check_case_exists(db, case_id)
    if case["status"] not in ("pending", "in_review"):
        raise BusinessError(f"case {case_id} 状态为 {case['status']}，不可审批")
    try:
        # INSERT action + UPDATE case 在同一事务
        db.execute(...)  # INSERT review_actions
        db.execute(...)  # UPDATE review_cases
        db.commit()
    except Exception as e:
        db.rollback()
        logger.error(f"[HITL] approve failed: case_id={case_id} {e}")
        raise AppError(500, "审批操作失败")
```

## 七、trace_id 注入方案

### 7.1 中间件注入

```python
# backend/middleware/trace.py
import uuid
from starlette.middleware.base import BaseHTTPMiddleware

class TraceMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        trace_id = request.headers.get("X-Trace-ID") or str(uuid.uuid4())[:8]
        request.state.trace_id = trace_id
        response = await call_next(request)
        response.headers["X-Trace-ID"] = trace_id
        return response
```

### 7.2 日志绑定

```python
# 在 TraceMiddleware 中绑定 loguru context
with logger.contextualize(trace_id=trace_id):
    response = await call_next(request)
```

## 八、日志策略

| 异常级别 | 日志级别 | 内容 |
|---------|---------|------|
| 400 BusinessError | `WARNING` | error_code + message + trace_id |
| 401/403 Auth | `WARNING` | error_code + user_info + trace_id |
| 404 NotFound | `INFO` 或 `WARNING` | error_code + 资源类型 + trace_id |
| 409 Conflict | `WARNING` | error_code + target + trace_id |
| 422 Validation | `WARNING` | 字段路径 + 错误详情 + trace_id |
| 429 RateLimit | `WARNING` | client_key + trace_id |
| 500 DatabaseError | `ERROR` | **完整异常链** + trace_id + SQL 上下文（脱敏） |
| 500 InternalError | `ERROR` + `exception()` | **完整堆栈** + trace_id |
| 503 Unavailable | `WARNING` | agent_name / 服务名 + trace_id |

## 九、实施计划（建议分 3 步）

### Step 1：全局兜底（最高优先级，~30 分钟）

1. 在 `core/exceptions.py` 中新增 `_error_response()` 函数
2. 新增 `http_exception_handler` 和 `unhandled_exception_handler`
3. 修改 `app_error_handler` 的响应结构，增加 `meta` 字段
4. 在 `main.py` 中注册 4 个 handler
5. 新增 `middleware/trace.py` 注入 trace_id

**效果**：所有异常响应统一为 `{code, data, message, meta}` 格式，生产环境不泄露堆栈。

### Step 2：异常子类化 + 错误码（中优先级，~1 小时）

1. 在 `core/exceptions.py` 中新增异常子类
2. 为 `AppError` 添加 `error_code` 字段（向后兼容）
3. 逐步将路由层的 `AppError(404, ...)` 改为 `ResourceNotFoundError(...)`
4. 将 `external/analyze.py` 和 `workflow.py` 的 `HTTPException` 改为 `AppError` 子类
5. 修复 P2 错误信息泄露点

**效果**：异常类型语义化，前端可按 `meta.error_code` 做差异化处理。

### Step 3：Repo 层 + Governance 层加固（低优先级，~1 小时）

1. HITL approve/edit/reject 增加 try/except + rollback
2. 关键 Repo 方法的异常处理策略评审
3. 评估是否需要引入 `RepoResult` 包装类型
4. hitl.py 的 `raise ValueError` 改为 `raise BusinessError`

**效果**：事务安全性提升，异常分层更清晰。

## 十、前端适配要点

### 10.1 Axios 拦截器调整

当前前端可能按 `response.data.code` 或 `error.response.status` 判断错误。改造后：

```javascript
// 统一错误处理
axios.interceptors.response.use(
  (res) => {
    // 所有 2xx 响应
    if (res.data.code && res.data.code !== 200) {
      // AppError 类错误（HTTP 200 但 body.code != 200）
      // 改造后不再出现此情况，所有错误 HTTP 状态码与 body.code 一致
    }
    return res;
  },
  (error) => {
    const { code, message, meta } = error.response?.data || {};
    // meta.error_code 可用于差异化处理
    switch (meta?.error_code) {
      case 'AUTH_FAILED':
        // 跳转登录
        break;
      case 'RATE_LIMITED':
        // 显示限流提示
        break;
      default:
        // 通用错误提示：使用 message 字段
        ElMessage.error(message || '请求失败');
    }
    return Promise.reject(error);
  }
);
```

### 10.2 关键变更

| 改前 | 改后 |
|------|------|
| 401/403 返回 `{"detail": "..."}` | 401/403 返回 `{code, data, message, meta}` |
| AppError 返回 HTTP 200 + body.code!=200 | AppError 返回 **真实 HTTP 状态码** + body.code 一致 |
| 无 `meta.error_code` | 新增 `meta.error_code` 字段 |

> **注意**：当前 `app_error_handler` 返回的 HTTP 状态码就是 `exc.code`，这点不变。但之前 `HTTPException` 的响应格式不同，改造后统一。

## 十一、风险与回滚

| 风险 | 缓解 |
|------|------|
| 前端依赖旧的 `{"detail": "..."}` 格式 | Step 1 完成后，前端需同步检查 401/403/422 的错误解析逻辑 |
| `AppError(code, message)` 旧调用签名 | 新增 `error_code` 参数设默认值，完全向后兼容 |
| `unhandled_exception_handler` 可能隐藏 bug | 仅生产环境隐藏 detail，开发环境保留完整信息 |
| SSE 流式响应不走全局 handler | SSE 端点内的 try/except 保持不变，这是正确的 |

**回滚方案**：Step 1 的核心改动集中在 `core/exceptions.py` + `main.py`，回退只需移除 handler 注册即可恢复原状。
