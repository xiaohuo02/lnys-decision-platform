## 二、各层异常处理详细分析

### 2.1 路由层（Routers）

#### 2.1.1 Business Routers（8 个）

| 文件 | 异常处理模式 |
|------|-------------|
| `customers.py`, `forecast.py`, `fraud.py`, `sentiment.py`, `inventory.py`, `association.py`, `dashboard.py` | **"瘦路由"无 try/except**，完全依赖 service 层和全局 handler |
| `chat.py` | SSE `event_generator` 内 `except Exception` → yield SSE error 事件 |
| `report.py` | `_fetch_module_summary` 捕获 Exception→返回 None；`download_report` 抛 `AppError(404)` |

#### 2.1.2 Admin Routers（14 个）

**常见模式：**
- **404 查询为空** → `raise AppError(404, "xxx不存在")`（traces, prompts, policies, knowledge, memory, audit, releases, reviews）
- **409 幂等/重复** → catch `IntegrityError` 或 `DuplicateActionError` → `AppError(409)`（prompts, policies, releases, evals）
- **400 业务校验** → catch `ValueError` from HITL → `AppError(400)`（reviews）

**问题点：**

| 问题 | 文件 | 详情 | 状态 |
|------|------|------|------|
| **降级吞异常** | `admin/dashboard.py:73` | `except Exception` → `degraded()` 零值，`str(e)` 仅在 logger 中，未返回前端 | ✅ 安全 |
| **降级吞异常** | `admin/agents.py:89` | `except Exception` → stats/errors/wfs 返回空 dict | ⚠️ 低风险 |
| ~~**错误信息泄露**~~ | `admin/ops_copilot.py:50` | ~~`str(e)` 直接写入 answer~~ → 现返回固定消息 `"运维助手暂时不可用"` | ✅ 已修复 |
| **分页 count 吞异常** | `reviews.py`, `prompts.py`, `policies.py` | count 查询 `except Exception` → fallback `len(items)` | ⚠️ 低风险 |
| **认证层混用** | `admin/auth.py:42` | DB 异常时 dev 后门放行，否则 `AppError(503)` | ⚠️ 设计意图，可接受 |

#### 2.1.3 External / Workflow / Internal Routers

| 文件 | 异常处理模式 | 问题 | 状态 |
|------|-------------|------|------|
| `external/analyze.py` | 已改用 `AppError` | ~~HTTPException 混用~~ → 路由层无 `raise HTTPException` | ✅ 已修复 |
| `external/chat.py` | 无 try/except，agent 异常冒泡 | 现有 `unhandled_exception_handler` 兜底，不会泄露堆栈 | ⚠️ 低风险 |
| `external/workflow_sse.py` | `except Exception` → yield SSE error | ~~含 `str(e)`~~ → 现返回固定消息 `"流式推送异常"` | ✅ 已修复 |
| `workflow.py` | 已改用 `AppError` | ~~HTTPException 混用~~ → 路由层无 `raise HTTPException` | ✅ 已修复 |
| `internal/smoke.py` | `except Exception` → `{"status":"error","error":str(e)}` | 内部路由，可接受 | ✅ |
| `internal/services.py` | 无 try/except，异常冒泡到全局 handler | 内部调试路由，`unhandled_exception_handler` 兜底 | ✅ |

### 2.2 Service 层

#### 2.2.1 通用模式（customer/forecast/sentiment/inventory/association/dashboard_service）

```
数据获取三级降级链:
  1. Redis 热缓存 → 命中返回 ok()
  2. CSV 文件读取 → 命中返回 ok()，同时回写 Redis+DB
  3. Mock 数据   → 返回 degraded()
  4. 全部失败    → raise AppError(503, "xxx暂未就绪")
```

**异常处理特征：**
- **Redis 操作**：全部 `try/except Exception` + `logger.warning`，静默失败不影响主流程 ✅
- **CSV 读取**：`try/except Exception` + `logger.warning`，失败后 fallthrough 到 Mock 层 ✅
- **DB 持久化**（通过 repo）：repo 层内部 `try/except` + rollback，返回 bool/None ✅
- **Agent 调用**：统一通过 `AgentGateway.call()` → 超时/异常/非 dict 均返回 None ✅

#### 2.2.2 特殊 Service

| Service | 特殊处理 |
|---------|---------|
| `fraud_service.py` | HITL 案例创建 `try/except`(non-fatal)；Redis lpush `try/except`(non-fatal)；repo insert `try/except`(non-fatal)；~~list_pending `AppError(500, f"...{e}")`~~ → 现为静态消息 `"待审核列表查询失败"` ✅ |
| `chat_service.py` | 全流程 Redis 操作静默失败；repo 落库静默失败；Agent 返回无 `reply` 键→fallback |
| `fraud_scoring_service.py` | 模型 lazy load 三路独立 `try/except`；评分三路独立 `try/except`→降级融合 ✅ |
| `report_rendering_service.py` | 磁盘写入 `try/except`(non-fatal) ✅ |
| `data_preparation_service.py` | CSV 读取 `try/except` → 返回 error 字段而非抛异常 ✅ |

### 2.3 数据访问层（Repositories）

| Repo | 异常处理 | 问题 |
|------|---------|------|
| `analysis_results_repo.py` | save: `try/except` + rollback→return False；get_latest: `try/except`→return None | 异常被完全吞没，上层无感知 |
| `chat_repo.py` | insert_pair/delete_session/get_history: 全部 `try/except` + rollback→return 默认值 | 同上 |
| `fraud_repo.py` | insert: `try/except` + rollback→return False；today_stats: `try/except`→return `{}` | 同上 |

**整体问题**：Repo 层把所有 DB 异常全部吞没为默认值，调用方无法区分"查无数据"和"查询出错"。

### 2.4 Governance 层

| 模块 | 异常处理 | 问题 |
|------|---------|------|
| `hitl.py` | `_check_case_exists` → `raise ValueError`；approve/edit/reject → `raise ValueError` for 状态校验 | 纯 DB 操作无 try/except，IntegrityError 等直接冒泡 |
| `action_ledger.py` | `raise DuplicateActionError`；其余 DB 操作无 try/except | 同上，DB 异常冒泡 |
| `audit.py` | `write_audit_log` 全包 `try/except`(non-fatal) | 审计写入失败不影响主流程 ✅ |
| `eval_service.py` | `raise ValueError` for 状态校验 | DB 操作无 try/except |

### 2.5 中间件 / 鉴权

| 模块 | 异常类型 | 问题 | 状态 |
|------|---------|------|------|
| `middleware/auth.py` | `HTTPException(401)` / `HTTPException(403)` | ~~不走 AppError handler~~ → 现由 `http_exception_handler` 统一转换为 `{code, data, message, meta}` | ✅ 已修复 |
| `core/rate_limiter.py` | `HTTPException(429)` | 同上 | ✅ 已修复 |

### 2.6 Agent Gateway

`AgentGateway.call()` 统一吞没所有 Agent 异常并返回 None：
- `asyncio.TimeoutError` → None
- `Exception` → None
- 非 dict 返回 → None
- Agent is None → None

**评价**：这是一个很好的边界隔离设计 ✅，但 **调用方无法区分"Agent 超时"和"Agent 内部错误"和"Agent 未加载"**。

## 四、异常处理模式统计

### 4.1 异常抛出方式分布（更新后）

| 方式 | 次数（约） | 文件分布 | 备注 |
|------|-----------|---------|------|
| `raise AppError(code, msg)` | ~30+ | admin routers, services | 主要方式，向后兼容 |
| `raise HTTPException(code, detail)` | ~3 | middleware/auth, rate_limiter | 仅中间件层，由 `http_exception_handler` 统一转换 |
| `raise ValueError(msg)` | ~6 | hitl.py, eval_service.py | 路由层 catch → AppError(400) |

> **变化**：路由层已不再有 `raise HTTPException`（从 ~10 降至 0），全部改用 `AppError`。仅 middleware 层保留 HTTPException。

### 4.2 try/except 模式分布

| 模式 | 次数（约） | 评价 |
|------|-----------|------|
| `except Exception` → log + 静默（返回默认值） | ~40+ | 过度使用，部分场景应区分异常类型 |
| `except Exception` → log + `raise AppError` | ~5 | 合理转换 |
| `except IntegrityError` → `raise AppError(409)` | ~3 | 精确捕获 ✅ |
| `except ValueError` → `raise AppError(400)` | ~3 | 精确捕获 ✅ |
| `except JWTError` → `raise HTTPException(401)` | ~1 | 精确捕获 ✅ |
| `except HTTPException: raise` + `except Exception` | ~1 | rate_limiter 正确做法 ✅ |

## 五、当前设计的优点

1. **AgentGateway 边界隔离**：Agent 层异常不会穿透到 HTTP 层 ✅
2. **三级降级链**（Redis→CSV→Mock→503）：数据服务层容错性好 ✅
3. **`ok()/degraded()/cached()` 统一包装函数**：成功响应格式统一 ✅
4. **审计日志非阻塞写入**：`write_audit_log` 失败不影响主流程 ✅
5. **Repo 层 rollback 习惯良好**：所有 catch 块都调用了 `db.rollback()` ✅
6. **`DuplicateActionError` 幂等设计**：高风险操作有幂等保护 ✅
7. **全局 4 层 Handler 覆盖**：`AppError` → `HTTPException` → `ValidationError` → `Exception` 兜底 ✅
8. **统一错误响应含 meta**：所有错误响应均包含 `{code, data, message, meta}` 结构 ✅
9. **异常继承树语义化**：10 个子类覆盖常见 HTTP 状态码，向后兼容旧 `AppError(code, msg)` ✅
10. **堆栈信息保护**：生产环境不暴露异常详情，仅 `DEV_BACKDOOR_ENABLED` 时通过 `meta.detail` 输出 ✅
