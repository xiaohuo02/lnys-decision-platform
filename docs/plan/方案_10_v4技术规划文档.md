## 一、规划目标

本项目从原来的“每个业务模块尽量 Agent 化”的方案，升级为 **v4.0：少量核心 Agent + 大量确定性服务 + 治理后台** 的工程化架构。

升级后的目标不是只做一个会话式多智能体 Demo，而是建设一套能够：

- 支撑经营分析与客户服务双场景
- 保留现有 `ml/` 目录中的模型资产和指标成果
- 与当前 `backend/`、`frontend/`、`docker-compose.yml` 协同演进
- 适合 `main / dev / feature/*` 的 Git 协作模式
- 能被拆成小任务持续交给 AI 辅助开发

## 二、关键决策：v4.0 如何适配当前仓库

### 2.1 不直接照搬原始 v4.0 方案的部分

为降低迁移风险，以下内容 **不直接按原始 v4.0 全量替换**：

- **不立即切换为 Monorepo 多应用结构**
  - 当前仓库已经稳定存在 `backend/`、`frontend/`、`ml/`
  - 现阶段采用“单仓渐进重构”，而不是一次性重排目录

- **不立即切换前端到 Next.js / shadcn/ui**
  - 当前前端是 `Vue 3 + Vite + Element Plus + ECharts`
  - 治理后台先在现有 `frontend/` 内增量实现，统一风格即可

- **业务主库不立即切换到 PostgreSQL**
  - 当前后端和 Docker 已围绕 MySQL 组织
  - v4.0 第一阶段继续使用 **MySQL + Redis** 作为业务与治理主存储
  - LangGraph checkpoint 独立使用 PostgreSQL，覆盖联调、演示与部署环境

- **不立即拆成多个独立部署微服务**
  - 当前先以 **FastAPI 模块化单体** 实现
  - 用目录隔离 `agents/`、`services/`、`governance/`、`routers/`

### 2.2 直接采用的 v4.0 核心思想

以下原则直接作为新的开发主线：

- **Agent 负责理解、路由、解释、收敛**
- **服务负责计算、规则、评估、公式与可复算结果**
- **状态只传摘要和引用，不在状态里传大块原始结果**
- **高风险操作必须经过 Guardrails / HITL**
- **所有 Prompt / Policy / Workflow 的变更都需要治理能力支持**

## 四、v4.0 的核心 Agent 设计

### 4.1 核心 Agent 清单

| Agent | 职责 | 是否直接跑 ML |
|------|------|---------------|
| `SupervisorAgent` | 识别请求、判断复杂度、选 workflow、分配服务/Agent | 否 |
| `InsightComposerAgent` | 读取多个 artifact，生成管理层可读摘要 | 否 |
| `OpenClawCustomerAgent` | 客服会话、意图识别、FAQ/订单查询、低置信转人工 | 否 |
| `RiskReviewAgent` | 汇总风控结果、输出人工审核摘要、记录复盘 | 否 |
| `OpsCopilotAgent` | 面向后台管理员的自然语言运维助手 | 否 |

### 4.2 关键原则

- Agent 不直接运行 XGBoost、LightGBM、LDA、EOQ 等底层逻辑
- Agent 通过调用服务、读取 artifact、解释结果来完成协作
- 只有真正需要自然语言理解、跨模块协调、总结生成的部分交给 LLM

## 五、业务服务层设计

### 5.1 服务清单

| 服务 | 主要能力 |
|------|---------|
| `DataPreparationService` | 数据读取、字段标准化、去重、缺失值处理、OMO 融合 |
| `CustomerIntelligenceService` | RFM、分群、流失预测、CLV、Cohort |
| `SalesForecastService` | Stacking、Prophet 对照、Holt-Winters 降级 |
| `FraudScoringService` | 规则引擎、LightGBM、AutoEncoder、融合评分 |
| `SentimentIntelligenceService` | 情感分类、主题挖掘、竞品对比、Granger |
| `InventoryOptimizationService` | ABC-XYZ、EOQ、安全库存、补货建议 |
| `AssociationMiningService` | FP-Growth / Apriori、推荐规则集 |
| `ReportRenderingService` | 报告渲染、图表、Markdown / PDF 导出 |

### 5.2 服务落地原则

- 每个服务必须提供 **确定性输入输出 schema**
- 每个服务必须可以 **脱离 Agent 独立测试**
- 每个服务输出必须落为 **artifact 或结构化对象**
- 服务失败时优先返回 **可解释的降级结果**，而不是静默失败

## 六、工作流设计

### 6.1 经营总览分析

```text
入口请求
  -> Supervisor 路由
  -> DataPreparationService
  -> 并行调用 Customer / Forecast / Sentiment / Fraud Summary
  -> Inventory 等待 Forecast
  -> InsightComposer 汇总
  -> ReportRendering 输出
```

### 6.2 高风险交易审核

```text
实时交易
  -> FraudScoringService
  -> 命中高风险阈值
  -> LangGraph interrupt()
  -> HITL Review Center
  -> RiskReviewAgent 整理复核摘要
  -> 审计与样本沉淀
```

### 6.3 OpenClaw 客服会话

```text
用户问题
  -> 输入 Guardrails
  -> OpenClaw 理解意图
  -> FAQ / 订单 / 物流 / 客户画像查询
  -> 低置信 or 高风险转人工
  -> 有写操作时再走 Policy / 审批
```

### 6.4 后台运维问答

```text
管理员问题
  -> OpsCopilot
  -> 读取 traces / evals / prompts / policies / releases
  -> 输出自然语言诊断
  -> 提供 drill-down 链接
```

## 七、状态、记忆与产物

### 7.1 状态模型

状态层只保留轻量信息：

- `request_id`
- `thread_id`
- `route_plan`
- `artifact_refs`
- `guardrail_results`
- `hitl_status`
- `errors`
- `final_summary`

### 7.2 产物层

所有关键业务结果都要保存为 artifact：

- 客户洞察结果
- 销售预测结果
- 舆情分析结果
- 库存优化建议
- 风控摘要
- 导出报告
- 评测快照

### 7.3 长期记忆

- **仅 OpenClaw 允许长期用户记忆**
- 其他业务模块只保存业务产物和平台状态，不保存用户长期会话记忆
- 长期记忆写入前必须经过 `extract -> validate -> dedupe -> risk_check -> TTL -> write`

## 九、Git 协作适配规则

### 9.1 分支模型

继续沿用当前项目的：

- `main`：稳定生产分支
- `dev`：集成联调分支
- `feature/*`：单任务功能分支

但从 v4.0 开始，**分支不再按传统模块粗粒度命名，而按阶段 / 工作流 / 治理能力命名**。

### 9.2 推荐分支命名

| 分支 | 适用场景 |
|------|---------|
| `feature/phase1-platform-skeleton` | 平台骨架、目录、基础治理表 |
| `feature/service-customer-intel` | 客户智能服务层 |
| `feature/service-forecast` | 销售预测服务层 |
| `feature/service-fraud-scoring` | 风控服务层 |
| `feature/agent-supervisor` | Supervisor Agent |
| `feature/workflow-business-overview` | 经营总览工作流 |
| `feature/workflow-risk-review` | 风控审核工作流 |
| `feature/openclaw-session` | 客服会话链路 |
| `feature/console-trace-center` | Trace 中心页面 |
| `feature/console-prompt-center` | Prompt Center 页面 |
| `feature/eval-center` | 评测中心 |

### 9.3 协作规则

- **一个任务 = 一个分支 = 一个 PR**
- 新能力开发前，优先更新 `方案_10_v4技术规划文档.md` / `方案_11_v4开发任务清单.md` / ADR
- PR 标题必须标注任务编号，例如：`feat(P1-T03): 新增 Supervisor 工作流骨架`
- 先合并文档和 schema，再合并 service / router / frontend 实现
- 不允许在同一分支里同时做大量无关修改

### 9.4 目录职责边界

| 目录 | 主要负责人 |
|------|-----------|
| `backend/agents/` | Agent 开发 |
| `backend/services/` | 后端 / 算法协同 |
| `backend/routers/` | 后端开发 |
| `backend/db/` | 后端开发 |
| `frontend/src/views/console/` | 前端开发 |
| `ml/` | 算法与模型训练 |
| `docs/plan/` `docs/architecture/` | 方案维护者 |

## 十二、给当前仓库的明确建议

### 12.1 现在就做

- 保留 `Vue 3 + FastAPI + MySQL + Redis`，并为 LangGraph 引入独立 PostgreSQL checkpoint
- 引入 `LangGraph` 作为工作流与 Agent 编排层
- 先实现模块化单体，不先拆微服务
- 在现有前端中增加治理后台，不先换前端框架
- 把所有 AI 开发任务拆成可 review 的 `方案_11_v4开发任务清单.md` 原子任务

### 12.2 暂缓做

- 暂缓将业务主库迁移到 PostgreSQL
- 暂缓整体改造成 Monorepo
- 暂缓独立部署多服务
- 暂缓大规模替换前端技术栈

这样可以最大限度复用你们现有仓库和已有代码，也最符合当前 Git 协作与 AI 辅助开发节奏。
