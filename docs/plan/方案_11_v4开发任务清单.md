> **真相源说明**：本文件是当前项目的开发执行清单  
> **对应主规划**：`docs/plan/方案_10_v4技术规划文档.md`  
> **架构依据**：`docs/architecture/01_前端架构决策.md` ~ `04_依赖与框架决策.md`  
> **执行原则**：一个任务 = 一个分支 = 一个 PR = 一个独立 AI 会话

---

## 一、任务执行规则

### 1.1 分支规则

所有任务必须从 `dev` 拉出独立分支，推荐命名：

- `feature/<task-id>-<short-name>`

示例：

- `feature/p1-t01-project-skeleton`
- `feature/p2-t03-workflow-business-overview`
- `feature/p3-t02-prompt-center-api`

### 1.2 提交规则

PR / Commit 建议带任务编号：

- `feat(P1-T03): 新增治理表 ORM 与初始化脚本`
- `feat(P2-T04): 接通 FraudScoringService 与 HITL 流程`
- `docs(P0-T01): 落地 v4.0 主规划与架构 ADR`

### 1.3 完成定义

每个任务完成时至少满足：

- 有明确代码落点
- 有可运行或可验证结果
- 有必要的文档或 schema 更新
- 不把无关修改混进同一个 PR
