## 一、现状分析与差距

| 维度 | 现状 | 目标 |
|---|---|---|
| 文档类型 | 仅手动输入纯文本 FAQ | PDF/Word/Excel/CSV/图片/扫描件 |
| 数据清洗 | 无清洗管线，原文直存 | 智能清洗：去噪→标准化→质量评分→入库 |
| OCR | 无 | 轻量 OCR 模型处理扫描件/图片 |
| 分块策略 | 固定 400 字窗口 | 语义感知递归分块 + 标题层级保留 |
| 向量存储 | ChromaDB 单 collection, 纯向量检索 | 多 collection 分域 + 混合检索(BM25+向量) |
| 文档生命周期 | doc→chunk 映射不完整，无版本管理 | 完整 doc→chunk→vector 溯源 + 软删除 + 版本 |
| 容错降级 | 无 fallback 体系 | 每层独立降级 + 全链路 fallback 链 |
| 安全防护 | 无上传校验/内容审查 | 文件校验→内容审查→PII检测→权限隔离 |
| 前端管理 | 仅 FAQ 列表 CRUD | 文件上传、处理进度、多知识库切换、检索测试 |

## 二、技术选型

| 组件 | 选型 | 理由 |
|---|---|---|
| OCR 引擎 | **RapidOCR** (rapidocr + onnxruntime) | 最轻量(~50MB)，ONNX推理无需PaddlePaddle重依赖，中英文准确率与PaddleOCR持平，CPU友好，Docker兼容好 |
| PDF 解析 | **pdfplumber** | 精准表格提取+文本定位，中文PDF友好，轻量无外部依赖 |
| Word 解析 | **python-docx** | 解析 .docx 段落/表格/标题层级 |
| Excel 解析 | **openpyxl** | 轻量，解析 .xlsx 为结构化行列数据 |
| 分块引擎 | 自研递归语义分块器 | 基于标题层级+段落边界+语义相似度断裂检测 |
| 向量存储 | **ChromaDB**(保持) | 项目规模(<100万向量)完全足够，已有集成 |
| 混合检索 | **BM25 (rank_bm25) + ChromaDB** + RRF融合 | 业界最佳实践：关键词+语义双路 |
| Embedding | **bge-small-zh-v1.5**(保持) | 512维，中文效果优秀，已部署 |
| 文件校验 | **python-magic** + 自研 | MIME类型真实检测，防伪装 |

## 五、核心模块详细设计

### 5.1 文件解析器 (file_parser.py)

**模式**: 策略模式，`{扩展名: Parser}` 映射表自动分发。

- **PDF**: pdfplumber 逐页提取文本+表格。每页文本 < 50字符且存在图片对象 → 判定扫描页 → 走 OCR。表格转 Markdown 格式保留结构。混合 PDF（部分文本页+部分扫描页）逐页判定分别处理。
- **DOCX**: python-docx 遍历 paragraphs + tables。通过 `paragraph.style.name` 识别 Heading 1/2/3 层级，构建标题树。内嵌图片提取后走 OCR。
- **XLSX**: openpyxl 逐 sheet 读取。列数 ≤ 6 转 Markdown 表格，> 6 转 key-value 逐行描述（避免宽表语义稀释）。
- **CSV**: pandas.read_csv + chardet 自动检测编码，处理逻辑同 XLSX。
- **TXT/MD**: 直接读取，保留 Markdown 标题标记。
- **图片**: 直接走 OCR 引擎。

统一输出 `ParseResult(raw_text, pages[], tables[], headings[], metadata{}, ocr_used, ocr_confidence)`。

### 5.2 OCR 引擎 (ocr_engine.py)

- 单例懒加载（与 EmbeddingService 同模式），首次调用才加载模型（~200ms）
- `ocr_image(file_bytes)` → 文本 + 每行置信度
- `ocr_pdf_page(pdf_path, page_num)` → 内部渲染为图片(300dpi)后调 ocr_image
- `is_scanned_pdf(pdf_path)` → 抽样前3页判定
- 置信度均值 < 0.6 → 标记 `quality_warning: "OCR置信度较低"`
- Docker 构建时预缓存模型到镜像，运行时无需联网

### 5.3 文本清洗器 (text_cleaner.py)

Pipeline 模式，有序步骤链，每步职责单一，可按知识库配置启用/禁用：

1. **编码标准化**: `unicodedata.normalize('NFC')`
2. **控制字符清除**: 正则去零宽字符/BOM/不可见字符
3. **页眉页脚去除**: 统计跨页重复行（出现率 > 70%），判定并移除
4. **页码清除**: 匹配 `^\s*\d{1,4}\s*$`、`- N -`、`第N页` 等模式
5. **多余空白合并**: 连续空行→单空行，行内连续空格→单空格
6. **断行修复**: 行尾非句终标点且下行首非标点 → 合并为同段
7. **表格格式保留**: 确保 Markdown 表格标记完整性
8. **特殊字符标准化**: 全角→半角标点（可配置）

### 5.4 质量评分器 (quality_scorer.py)

四维加权评分（0-1）：

| 维度 | 权重 | 计算方法 |
|---|---|---|
| 完整性 | 0.30 | 非空行占比；全文 < 20字符 → 0分 |
| 可读性 | 0.25 | 乱码字符比（CJK+ASCII+常用标点之外）；平均句长合理性(20-80字符) |
| 信息密度 | 0.25 | jieba分词去停用词后有效词占比；大量孤立单字 → 扣分 |
| 结构性 | 0.20 | 是否有标题层级；段落长度合理性(50-500字符)；有无列表/表格 |

阈值：≥ 0.6 直接入库 | 0.3-0.6 入库+警告 | < 0.3 拒绝(status=failed)，原文保留可人工编辑重处理。

### 5.5 分块引擎 (chunk_engine.py)

三级递归分块：

- **Level 1**: 按标题（H1/H2/H3）分大段
- **Level 2**: 段内按段落/双换行分中段
- **Level 3**: 超长段按句子边界分小段

特殊处理：表格整表=一个chunk | 代码块整块保留 | 连续列表项聚合。

每个 chunk 前面附加 `heading_path`（标题路径），提升向量化后的检索精度。

默认参数：max_tokens=512 | min_tokens=50(防碎片) | overlap_tokens=64(~12%) | heading_attach=true。

### 5.6 混合检索引擎 (search_engine.py)

双路检索 + RRF 融合：

1. **向量路**: EmbeddingService → ChromaDB.query(top_k*2)
2. **BM25路**: jieba分词 → rank_bm25.BM25Okapi(top_k*2)
3. **RRF融合**: `score = Σ 1/(k+rank_i)`，k=60
4. **元数据过滤**: lib_id/domain/file_type/quality_score
5. **去重**: 同doc_id只保留最高分chunk
6. **返回 top_k**

BM25 索引按知识库懒加载 + LRU缓存(最近5个)，文档变更时标记脏位，下次检索自动重建。

## 六、多级 Fallback 容错体系

### 6.1 设计原则

- **每一层独立降级**：任何一个组件故障不应导致整个知识库不可用
- **降级有感知**：所有降级操作都在响应 `meta.degraded` + `meta.warnings` 中标记，前端可展示降级提示
- **静默不丢数据**：处理失败的文档保留原文和错误信息，可随时重处理
- **审计可追溯**：所有降级事件写入 trace_center

### 6.2 文档处理管线 Fallback 链

```
上传文件
  │
  ├─ FileParser 失败
  │    ├─ PDF解析失败 → 尝试 OCR 整个文件（当作扫描件处理）
  │    ├─ DOCX解析失败 → 尝试用 python-magic 检测真实类型重新分发
  │    ├─ XLSX解析失败 → 尝试 csv fallback（部分xlsx实为csv另存）
  │    └─ 全部失败 → status='failed', error_msg记录, 原文件保留
  │         → 前端展示"解析失败，支持手动输入文本"选项
  │
  ├─ OCR 失败
  │    ├─ RapidOCR 模型未加载 → 跳过 OCR，仅用文本提取结果
  │    ├─ 单页 OCR 超时(>30s) → 跳过该页，标记 warning
  │    ├─ OCR 置信度 < 0.3 → 丢弃该页 OCR 结果，标记 warning
  │    └─ 全页均失败 → 仅保留文本提取部分，quality_score 扣分
  │
  ├─ TextCleaner 失败
  │    └─ 任何清洗步骤异常 → 跳过该步骤继续后续步骤
  │         → 最坏情况：raw_text 直接作为 cleaned_text 使用
  │
  ├─ QualityScorer 失败
  │    └─ 评分异常 → 默认给 0.5 分，标记 warning:"质量评分不可用"
  │
  ├─ ChunkEngine 失败
  │    ├─ 递归分块失败 → 回退到固定窗口分块(400字+100重叠)
  │    └─ 固定分块也失败 → 全文作为单个 chunk 入库
  │
  ├─ EmbeddingService 失败
  │    ├─ 模型未加载 → status='failed', 保留 chunks 在 MySQL
  │    │    → 定时任务重试(每5分钟检查 status='failed' 的文档)
  │    └─ 单批 embed 超时 → 缩小批次重试(batch_size 减半)
  │
  └─ ChromaDB 写入失败
       ├─ 单次 upsert 失败 → 重试3次(指数退避: 1s/2s/4s)
       ├─ collection 不存在 → 自动创建后重试
       └─ 全部失败 → chunks 保留在 MySQL(chromadb_id 为空)
            → 下次 sync 或 reprocess 时补写
```

### 6.3 检索 Fallback 链

```
用户查询
  │
  ├─ Level 1: 混合检索 (Vector + BM25 + RRF)
  │    → 正常返回结果
  │
  ├─ Level 2: 仅向量检索 (BM25不可用时)
  │    触发条件: BM25 索引构建失败或内存不足
  │    → 仅用 ChromaDB 向量检索, meta.warnings 标记
  │
  ├─ Level 3: 仅 BM25 检索 (ChromaDB/Embedding 不可用时)
  │    触发条件: ChromaDB 连接失败或 Embedding 模型未加载
  │    → 仅用 BM25 关键词检索, meta.degraded=true
  │
  ├─ Level 4: MySQL LIKE 检索 (向量+BM25 都不可用时)
  │    触发条件: ChromaDB 和 BM25 索引均不可用
  │    → SELECT content FROM kb_chunks WHERE content LIKE '%关键词%'
  │    → meta.degraded=true, meta.source='mysql_like'
  │
  └─ Level 5: 返回空结果 + 明确提示
       触发条件: MySQL 也查询失败
       → {items: [], total: 0, meta: {degraded: true, warnings: ["知识库服务暂时不可用"]}}
```

### 6.4 Agent 调用 Fallback

```
Agent 请求知识库检索
  │
  ├─ SearchEngine.search() 正常
  │    → 返回检索结果供 Agent 使用
  │
  ├─ SearchEngine 超时 (>5s)
  │    → 返回空结果 + degraded 标记
  │    → Agent 使用自身上下文/历史回答（不依赖知识库）
  │
  ├─ SearchEngine 异常
  │    → 捕获异常，日志记录
  │    → Agent 回复 "暂时无法检索知识库，基于已有信息回答"
  │
  └─ OpenClaw 专用 Fallback
       ├─ 知识库检索无结果 → SQL LIKE 查 faq_documents(旧表)
       ├─ 旧表也无结果 → 回复 "暂时无法解答，正在转接客服"
       └─ handoff=true, 转人工

```

### 6.5 基础设施 Fallback

| 组件 | 故障场景 | Fallback |
|---|---|---|
| ChromaDB | 文件损坏/磁盘满 | 检索降级到 MySQL LIKE；入库队列化到 Redis 等恢复后重试 |
| Embedding 模型 | 模型文件缺失/内存不足 | 启动时 try/except 保护不崩；新文档标记 pending 等模型恢复 |
| OCR 模型 | 模型未下载/推理异常 | 跳过 OCR 步骤，仅使用文本提取；图片类文档标记 failed |
| MySQL | 连接池耗尽 | 读操作走 Redis 缓存；写操作排队重试 |
| Redis | 不可用 | BM25 索引降级为内存字典(不持久化)；频率限制暂时放行 |

## 七、安全防护与用户不确定性处理

### 7.1 文件上传安全（纵深防御）

#### 第一道：前端校验（快速拦截）
- 文件扩展名白名单: `.pdf/.docx/.xlsx/.csv/.txt/.md/.jpg/.png/.bmp/.tiff`
- 单文件大小限制: 50MB（前端 el-upload 的 `before-upload` 钩子）
- 单次批量上传限制: 最多 20 个文件
- 禁止空文件上传

#### 第二道：后端校验（严格验证）
- **MIME 类型真实检测**: 使用 `python-magic` 读取文件头魔术字节判断真实类型，**拒绝扩展名与真实类型不匹配的文件**（防 `.pdf` 实为 `.exe`）
- **文件大小二次校验**: 服务端限制 50MB，防止前端校验被绕过
- **文件名安全化**: 去除路径穿越字符（`../`、`\`、null 字节），截断超长文件名（>200字符），仅保留安全字符
- **SHA-256 哈希去重**: 计算文件指纹，重复文件提示"已存在"并给出已有文档链接
- **频率限制**: 每用户每分钟最多上传 10 个文件（基于现有 rate_limiter）

#### 第三道：内容安全（处理阶段）
- **恶意内容检测**: PDF 中的 JavaScript/宏/外部链接自动剥离（pdfplumber 仅提取文本层，天然过滤）
- **DOCX 宏检测**: 检测 `.docm` 或含宏的 `.docx`，标记 warning 但不阻止（只提取文本，不执行宏）
- **文件大小 vs 文本比**: 文件 > 10MB 但提取文本 < 100字符 → 标记"疑似无效文件"

### 7.2 PII（个人身份信息）检测

在 TextCleaner 后、入库前增加 PII 检测步骤：

| PII 类型 | 检测方法 | 处理方式 |
|---|---|---|
| 手机号 | 正则 `1[3-9]\d{9}` | 记录位置，中间4位替换为 `****` |
| 身份证号 | 正则 `\d{17}[\dXx]` + 校验位验证 | 记录位置，中间8位替换为 `****` |
| 银行卡号 | 正则 `\d{16,19}` + Luhn校验 | 记录位置，中间替换 |
| 邮箱 | 正则标准邮箱格式 | 记录但不脱敏（通常非高风险） |
| 地址 | jieba + 地址关键词检测（省/市/区/街道/号） | 标记 warning，不自动脱敏 |

- 检测结果写入 `kb_documents.pii_detected` 和 `pii_fields`
- **不自动删除**，仅标记 + 部分脱敏，由管理员决定是否继续入库
- 前端文档详情页展示 PII 警告标记

### 7.3 用户不确定性交互的安全方案

#### 场景 1：用户上传了不支持的文件类型
```
前端: 文件扩展名不在白名单 → 立即弹窗提示支持的格式列表
后端: MIME 类型不在白名单 → 返回 400 + 明确错误信息 + 支持格式列表
```

#### 场景 2：用户上传了损坏的文件
```
FileParser 解析失败 → status='failed'
前端: 展示"文件可能已损坏"提示
操作: 提供"手动输入文本"选项 + "重新上传"按钮
审计: 记录失败原因到 audit_log
```

#### 场景 3：用户上传了内容极少的文件
```
QualityScorer 评分 < 0.3 → status='failed'
前端: 展示"内容不足，无法有效向量化"
操作: 提供"查看提取文本"选项，用户可编辑补充后重新处理
```

#### 场景 4：用户上传了包含 PII 的文件
```
PII 检测发现敏感信息 → pii_detected=true
前端: 文档卡片显示 ⚠️ 标记 + 具体 PII 类型
操作: 管理员确认后可选择:
  a) 脱敏后入库（默认）
  b) 原文入库（需二次确认 + 审计记录）
  c) 拒绝入库
```

#### 场景 5：用户上传了重复文件
```
file_hash 匹配到已有文档
前端: 弹窗提示"该文件已存在于知识库 X 中"
操作:
  a) 查看已有文档
  b) 强制上传为新版本（version+1，旧版本保留）
  c) 取消上传
```

#### 场景 6：用户删除了正在被 Agent 使用的文档
```
删除请求 → 检查该文档是否在近 1 小时内被检索引用
如果有引用:
  前端: 弹窗"该文档近期被使用，确认删除？"
  操作: 二次确认后执行软删除(status='disabled')，30天后自动清理
如果无引用:
  直接执行级联删除
```

#### 场景 7：用户在知识库为空时发起检索
```
SearchEngine 检测到目标知识库 chunk_count=0
前端: 返回空结果 + 提示"该知识库暂无文档，请先上传"
不降级、不报错，明确引导用户下一步操作
```

#### 场景 8：用户检索结果全部低分（最高分 < 0.3）
```
SearchEngine 返回结果但分数极低
前端: 展示结果但标记"相关度较低" + 建议:
  - "尝试换个关键词"
  - "该问题可能不在当前知识库范围内"
Agent 场景: 不使用低分结果，走自身知识/转人工
```

| 检查项 | 方法 | 健康标准 |

| 指标 | 采集方式 | 告警阈值 |
|---|---|---|

## 十七、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| RapidOCR 首次下载需联网 | 中 | Docker构建失败 | Dockerfile build 阶段预缓存；或提前下载挂载 volume |
| 大文件处理阻塞主线程 | 低 | API 响应慢 | BackgroundTasks 异步处理 + SSE/轮询进度 |
| BM25 索引内存占用过大 | 低 | OOM | LRU 缓存限制(5个库)；单库 > 10万chunk 时告警 |
| python-magic 在 Windows 需要额外 DLL | 中 | 本地开发不兼容 | 开发环境用 `python-magic-bin`；Docker(Linux) 无此问题 |
| 向后兼容 faq_documents 双写一致性 | 中 | 数据不一致 | 事务包裹双写；sync 接口可手动修复 |
| ChromaDB 单机性能上限 | 低 | 检索变慢 | 当前规模足够；超100万向量可迁移 Milvus Lite |
| OCR 对低质量扫描件识别差 | 中 | 入库文本质量低 | quality_scorer 拦截 + warning 提示人工审核 |
| PII 检测误报/漏报 | 中 | 合规风险 | 仅标记不自动删除；管理员二次确认 |
