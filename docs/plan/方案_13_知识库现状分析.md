## 一、整体架构总览

当前系统有 **两套独立的知识库**，共享同一套底层基础设施：

```
                    ┌─────────────────────────────────────────┐
                    │            消费方 (上层调用者)             │
                    ├───────────────┬─────────────────────────┤
                    │  KBRagSkill   │  OpenClaw Agent          │
                    │ (Copilot技能) │  (客服Agent)              │
                    │ 联合检索两库   │  优先向量→降级SQL LIKE     │
                    └───────┬───────┴──────────┬──────────────┘
                            │                  │
              ┌─────────────▼──────┐   ┌───────▼──────────────┐
              │ SentimentKBService │   │ EnterpriseKBService   │
              │   (舆情知识库)      │   │   (企业FAQ知识库)      │
              │                    │   │                       │
              │ collection:        │   │ collection:           │
              │ sentiment_reviews  │   │ enterprise_documents  │
              └─────────┬──────────┘   └──────────┬───────────┘
                        │                         │
              ┌─────────▼─────────────────────────▼───────────┐
              │            共享基础设施层                        │
              ├────────────────────┬───────────────────────────┤
              │  EmbeddingService  │  VectorStoreManager       │
              │  (bge-small-zh     │  (ChromaDB PersistentClient│
              │   v1.5, 512维)     │   cosine 距离)             │
              └────────────────────┴───────────────────────────┘
                        │                         │
                   models/                   chroma_data/
                bge-small-zh-v1.5          (持久化文件存储)
```

**核心理解**：这不是一个"知识库系统"，而是 **两个各自独立的 Service + 一层共享的 Embedding/ChromaDB 基础设施**。两者之间没有统一的管理、没有共享的检索引擎、没有统一的文档生命周期。

## 二、共享基础设施层

### 2.1 EmbeddingService（向量编码服务）

**文件**: `backend/core/embedding.py` (62行)

| 特性 | 实现 |
|---|---|
| 模式 | 线程安全单例（双重检查锁） |
| 模型 | BAAI/bge-small-zh-v1.5, 本地加载，33M参数 |
| 维度 | 512 |
| 资源 | ~200MB RAM, ~15ms/句(CPU) |
| 接口 | `embed(texts)` 批量 / `embed_one(text)` 单条 / `embed_query(query)` 查询 |
| 归一化 | `normalize_embeddings=True`（输出向量已L2归一化，cosine距离=点积） |

**理解要点**：两个知识库共用同一个模型实例，只加载一次。配置项 `BGE_MODEL_NAME` 指向 `models/bge-small-zh-v1.5` 本地目录。

### 2.2 VectorStoreManager（ChromaDB 管理器）

**文件**: `backend/core/vector_store.py` (104行)

| 特性 | 实现 |
|---|---|
| 模式 | 线程安全单例 |
| 存储 | ChromaDB PersistentClient → `chroma_data/` 目录 |
| Collection 隔离 | 通过命名前缀隔离 `sentiment_*` 和 `enterprise_*` |
| 距离度量 | cosine（`hnsw:space: cosine`） |

**Collection 注册表**（COLLECTION_REGISTRY，当前4个）：

| Collection 名 | Owner | 用途 |
|---|---|---|
| `sentiment_reviews` | sentiment | 每条评论的分析结果 embedding |
| `sentiment_entities` | sentiment | 按实体聚合的情感画像（已注册但未使用） |
| `sentiment_summaries` | sentiment | 周期性聚合摘要（已注册但未使用） |
| `enterprise_documents` | enterprise | FAQ 文档 embedding |

**理解要点**：VectorStoreManager 只是一个 ChromaDB 客户端的包装。它提供 `get_collection(name)` → 如果 collection 不存在就创建。注册表是"文档性质"的，不会阻止创建未注册的 collection（只是打个 warning）。

## 三、知识库 A — 企业 FAQ 知识库

### 3.1 数据来源

| 来源 | 表 | 说明 |
|---|---|---|
| MySQL | `faq_documents` | 手动录入的 FAQ 问答对 |

`faq_documents` 表结构：
- `doc_id` (UUID) — 唯一标识
- `group_name` — 分组: order/refund/shipping/account/general
- `title` — FAQ 标题（问题）
- `content` — FAQ 内容（答案），LONGTEXT
- `source` — 来源说明
- `is_active` — 是否启用(0/1)
- `created_by` — 创建者
- 种子数据: 16条 FAQ（见 `db/seed_knowledge.sql`）

### 3.2 Service 实现

**文件**: `backend/services/enterprise_kb_service.py` (258行)

#### 入库流程 (`ingest`)
```
输入: doc_id, title, content, group_name
  │
  ├─ 1. 先删除该 doc_id 的旧 chunk（防重复）
  ├─ 2. 标题+内容合并: "title\ncontent"
  ├─ 3. 固定窗口分块: 400字/块, 100字重叠
  │     └─ 短文本(≤400字) → 不分块，直接作为单 chunk
  ├─ 4. 为每个 chunk 生成 ID: "{doc_id}__chunk_{i}"
  ├─ 5. 批量 embed（调 EmbeddingService.embed）
  └─ 6. ChromaDB.upsert(ids, embeddings, documents, metadatas)
```

Metadata 格式：
```json
{
  "doc_id": "uuid",
  "chunk_index": 0,
  "title": "FAQ标题",
  "group_name": "order",
  "ingested_at": "2026-04-09T21:00:00"
}
```

#### 检索流程 (`search`)
```
输入: query, top_k=3, group_name(可选), min_similarity=0.45
  │
  ├─ 1. query → EmbeddingService.embed_query → 512维向量
  ├─ 2. ChromaDB.query(query_embeddings, n_results=top_k, where=group过滤)
  ├─ 3. cosine distance → similarity (1-distance)
  ├─ 4. 过滤 similarity < 0.45 的结果
  ├─ 5. 按 doc_id 去重（同文档多 chunk 只保留最高分）
  └─ 返回: [{doc_id, title, content, group_name, similarity}]
```

#### 全量同步 (`sync_all`)
```
MySQL faq_documents (WHERE is_active=1) → 逐条调 ingest → ChromaDB
返回: {total, ingested, failed}
```

#### 删除 (`remove`)
```
ChromaDB.get(where={"doc_id": xxx}) → ChromaDB.delete(ids)
```

### 3.3 管理 API

**文件**: `backend/routers/admin/knowledge.py` (175行)

| 方法 | 路径 | 功能 | 实现状态 |
|---|---|---|---|
| GET | `/admin/knowledge/faqs` | FAQ 列表(分页+筛选) | ✅ 完整 |
| GET | `/admin/knowledge/faqs/{id}` | FAQ 详情 | ✅ 完整 |
| POST | `/admin/knowledge/faqs` | 新建 FAQ | ✅ MySQL写入 + 异步ChromaDB同步 |
| POST | `/admin/knowledge/faqs/{id}/disable` | 停用 FAQ | ✅ MySQL标记 + 异步ChromaDB删除 |
| POST | `/admin/knowledge/sync` | 全量同步 | ✅ MySQL→ChromaDB |
| GET | `/admin/knowledge/kb-stats` | 统计信息 | ✅ chunk数/模型/维度 |

**ChromaDB 同步机制**：新建/停用 FAQ 时通过 FastAPI `BackgroundTasks` 异步执行入库/删除，不阻塞 HTTP 响应。

### 3.4 前端

| 文件 | 功能 |
|---|---|
| `frontend/src/api/admin/knowledge.js` | 4个API调用: getFAQs/getFAQ/createFAQ/disableFAQ |
| `frontend/src/adapters/knowledge.js` | 数据转换: toFAQList/toFAQDetail |

当前前端功能：FAQ 列表查看、新建 FAQ、停用 FAQ。**无文件上传、无检索测试、无多知识库管理**。

## 四、知识库 B — 舆情知识库

### 4.1 数据来源

| 来源 | 说明 |
|---|---|
| 舆情分析 Agent | 分析评论后自动写入（非手动录入） |

不依赖 MySQL 表，数据**直接写入 ChromaDB**（`sentiment_reviews` collection）。

### 4.2 Service 实现

**文件**: `backend/services/sentiment_kb_service.py` (240行)

#### 入库流程 (`ingest`)
```
输入: review_id, text(原始评论), result(分析结果dict)
  │
  ├─ 1. text → embed_one → 512维向量
  ├─ 2. 构建 metadata:
  │     - label(情感标签), confidence, model_used, cascade_tier
  │     - entity_sentiments → JSON字符串
  │     - intent_tags → JSON字符串
  │     - agent_signals → JSON字符串
  │     - entities → 逗号分隔实体名(用于过滤)
  └─ 3. ChromaDB.upsert(review_id, embedding, text, metadata)
```

**注意**：每条评论=一个向量，**不分块**（评论通常短文本）。

#### 检索

| 方法 | 功能 |
|---|---|
| `search_similar(query, top_k, label)` | 语义相似评论检索，可按情感标签过滤 |
| `search_by_entity(entity, days, top_k)` | 按实体+时间范围检索，语义+元数据双过滤 |

### 4.3 管理 API

**无独立管理 API**。舆情知识库由 Agent 自动管理，不通过管理后台操作。

## 五、消费方 — 谁在用知识库

### 5.1 KBRagSkill (Copilot 技能)

**文件**: `backend/copilot/skills/kb_rag_skill.py` (126行)

**工作方式**：联合检索两个知识库

```
用户查询
  │
  ├─ 1. EnterpriseKBService.search(query, top_k)  → 企业FAQ结果
  ├─ 2. SentimentKBService.search_similar(query, top_k)  → 舆情结果
  ├─ 3. 统一格式化: {source, content, score, metadata}
  ├─ 4. 按 score 降序排序，取 top_k
  └─ 5. 发送 CopilotEvent (ARTIFACT + SUGGESTIONS + TOOL_RESULT)
```

**特点**：两库结果混合排序，不区分来源优先级。任何一库异常 → `except: pass`，静默跳过。

### 5.2 OpenClaw 客服 Agent

**文件**: `backend/agents/openclaw_agent.py`

**工作方式**：两级 FAQ 查询 fallback

```
用户消息
  │
  ├─ 1. 意图分类 → 确定是否需要查 FAQ
  │
  ├─ 2a. _vector_faq_search(query)  → 向量语义检索
  │     调用 EnterpriseKBService.search()
  │     有结果 → 使用 best.content 构造回复
  │
  ├─ 2b. 向量检索无结果/失败 → _faq_lookup(db, query)
  │     SQL LIKE 模糊查询 faq_documents 表
  │     有结果 → 使用 best.content 构造回复
  │
  └─ 3. 全部无结果 → "暂时无法解答，正在为您转接客服"
```

**特点**：
- OpenClaw 只使用企业 FAQ 知识库，不使用舆情知识库
- 有明确的两级降级：向量检索 → SQL LIKE → 转人工
- `_vector_faq_search` 内部处理了事件循环兼容（检测是否有 running loop，兼容同步/异步上下文）

## 九、一句话总结

**当前知识库 = 2个独立 Service（企业FAQ + 舆情）+ 共享 Embedding/ChromaDB 底座 + 纯文本手动录入 + 纯向量检索。能跑，但缺乏多格式支持、智能清洗、混合检索、统一管理、容错降级，距离企业级/比赛级有明显差距。**
