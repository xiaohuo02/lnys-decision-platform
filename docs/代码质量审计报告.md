## 二、亮点（值得保持的优秀实践）

### 2.1 优雅降级链 ✅
```
Agent → ML Service → 规则引擎 → Mock 数据 → 明确错误
```
每个 Service 都实现了 `Redis缓存 → DB持久化 → CSV文件 → Mock → 错误提示` 的完整降级链。
- `backend/agents/gateway.py` 的 AgentGateway 统一屏蔽 None/超时/异常三类问题
- `backend/core/error_handler.py` 的 ErrorClassifier 按异常类型智能分类处理策略（429限流→退避重试、认证失败→中止、超时→指数退避）
- 这种设计 **优于大多数开源项目**，对标 LangGraph 生产最佳实践

### 2.2 瘦路由 + 胖服务 ✅
- Router 层仅做参数校验和依赖注入，零业务逻辑
- Service 层承担全部业务编排
- 完全符合 [fastapi-best-practices](https://github.com/zhanymkanov/fastapi-best-practices) 推荐的分层原则

### 2.3 Agent 契约文档 ✅
- `supervisor_agent.py` 顶部的 Agent 契约框（输入/输出/允许调用/不能做/失败降级/HITL/依赖/Trace）是非常好的实践
- 建议所有 Agent 统一采用此契约模板

### 2.4 统一响应结构 
- `core/response.py` 提供 `ok()` / `cached()` / `error()` / `degraded()` / `not_ready()` 统一响应格式
- 所有响应均包含 `meta` 字段：`{degraded, source, trace_id, warnings}`（2026-04-03 增强）
- 前端 `request.js` 拦截器自动将 `meta` 附加为 `data._meta`（不可枚举属性）—— `requestBusiness` 和 `requestAdmin` 现已支持（2026-04-05 统一）
- 业务组件通过 `d?._meta?.degraded` 统一检测降级状态，替代旧的 `_degraded` 字段

### 2.5 配置管理 
- Pydantic Settings v2 + `model_validator` 自动构造派生 URL
- 生产环境强制校验 SECRET_KEY 不为默认值
- 环境变量优先级清晰：显式设置 > 组件字段自动构造

### 2.6 前端工程化 ✅
- Vue 3 + Pinia + Element Plus + ECharts 技术选型合理
- Vite 的 `manualChunks` 分包策略正确（vue/element/echarts 分离）
- 路由守卫完整（认证 → 初始化 → 角色权限 → 放行）
- 主题系统设计良好（localStorage 持久化 + 布局默认值 + 用户偏好优先）

## 三、问题发现与改进建议

### 3.1 🔴 安全性问题（高优先级）

#### 问题 S1：开发模式硬编码后门
**文件**：`backend/routers/admin/auth.py:49-58`、`backend/middleware/auth.py:57-59`

```python
# auth.py — 开发环境允许 admin/admin 登录
if settings.ENV == "development" and body.username == "admin" and body.password == "admin":
    token = create_access_token("admin", ["platform_admin"])

# middleware — 无 token 时返回 dev_admin
if settings.ENV == "development":
    return CurrentUser(username="dev_admin", roles=["platform_admin"])
```

**风险**：如果 ENV 变量在生产环境被误设为 `development`，攻击者可无密码获取管理员权限。

**建议**：
```python
# 1. 添加独立的 DEV_BACKDOOR_ENABLED 开关，默认 False
DEV_BACKDOOR_ENABLED: bool = False

# 2. 开发后门与 ENV 解耦
if settings.DEV_BACKDOOR_ENABLED and not settings.is_production:
    ...

# 3. 生产环境的 model_validator 中强制校验
if self.is_production and self.DEV_BACKDOOR_ENABLED:
    raise ValueError("DEV_BACKDOOR_ENABLED must be False in production")
```

#### 问题 S2：`.env.docker` 中包含明文密码且已提交到 Git
**文件**：`.env.docker:19,32,47,64`

```
DB_PASSWORD=lnys_password
POSTGRES_PASSWORD=lnys_pg_dev_password
SECRET_KEY=dev_only_secret_key_not_for_production_32c
MYSQL_ROOT_PASSWORD=rootpassword
```

**建议**：
- `.env.docker` 仅保留占位符，实际密码通过 `.env.docker.local`（gitignored）覆盖
- 或使用 Docker secrets / Vault 管理敏感信息
- 至少在 `.gitignore` 中排除所有 `.env*` 文件（当前 `.env.production` 已排除，但 `.env.docker` 未排除）

#### 问题 S3：JWT SECRET_KEY 默认值过弱
**文件**：`backend/config.py:56`

```python
SECRET_KEY: str = "change_this_in_production"
```

**建议**：开发环境也建议使用随机生成的密钥。`model_validator` 已检查生产环境，但开发环境的默认密钥也应更复杂：
```python
import secrets
SECRET_KEY: str = secrets.token_urlsafe(32)  # 每次启动随机
```

#### 问题 S4：缺少 Rate Limiting 中间件
当前 API 无请求频率限制，容易遭受暴力破解和 DDoS。

**建议**：添加 `slowapi` 或自定义 Redis-based 限流：
```python
from slowapi import Limiter
limiter = Limiter(key_func=get_remote_address)
app.state.limiter = limiter

@router.post("/auth/login")
@limiter.limit("5/minute")
async def login(request: Request, ...):
```

### 3.2 🟠 架构与代码质量（中优先级）

#### 问题 A1：同步 ORM + 异步路由的线程安全风险
**文件**：`backend/database.py`

当前使用同步 SQLAlchemy Engine，但路由是 `async def`。虽然 FastAPI 会在线程池中运行同步 `Depends`，但 Service 层混合使用了同步 DB 操作和异步 Redis 操作。

**对标最佳实践**（fastapi-best-practices）：
> "Set tests client async from day 0" — 推荐从一开始就使用异步数据库驱动

**建议**：
- 短期：确保所有 DB 操作都通过 `run_in_threadpool` 包装（当前 `check_db_health` 已做到，但 Service 层的 `self._repo.save()` 等未包装）
- 长期：迁移到 `sqlalchemy[asyncio]` + `aiomysql`
```python
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
engine = create_async_engine("mysql+aiomysql://...")
```

#### 问题 A2：Repository 层使用原始 SQL 而非 ORM Model
**文件**：`backend/repositories/analysis_results_repo.py`、`backend/routers/admin/auth.py`

```python
# 原始 SQL 拼接
self.db.execute(text("INSERT INTO analysis_results ..."))
row = db.execute(sqlalchemy.text("SELECT ... FROM users WHERE username = :u"))
```

**建议**：
- 定义 SQLAlchemy ORM Model（当前 `backend/models/` 为空目录！）
- 使用 ORM 查询代替原始 SQL，获得类型安全和自动参数化
- 参考 fastapi-best-practices：*"SQL-first, Pydantic-second"* 虽推荐 SQL 做复杂查询，但基础 CRUD 应使用 ORM

```python
# 推荐
class AnalysisResult(Base):
    __tablename__ = "analysis_results"
    id = Column(Integer, primary_key=True)
    module = Column(String(50))
    result_json = Column(Text)
    generated_at = Column(DateTime, server_default=func.now())

# Repository
def save(self, module: str, result: dict):
    obj = AnalysisResult(module=module, result_json=json.dumps(result))
    self.db.add(obj)
    self.db.commit()
```

#### 问题 A3：Service 层 Mock 数据硬编码
**文件**：`backend/services/customer_service.py:174-197`、`fraud_service.py:55-59`

每个 Service 内部都有大量硬编码的 Mock 数据块，导致：
- Service 文件膨胀
- Mock 数据分散在各处，难以统一维护

**建议**：
```python
# 1. 抽取 Mock 数据到独立模块
# backend/mock/customer_mock.py
MOCK_RFM = [...]
MOCK_SEGMENTS = {...}

# 2. Service 中引用
from backend.mock.customer_mock import MOCK_RFM
if settings.ENABLE_MOCK_DATA:
    return degraded(MOCK_RFM, "mock data")
```

#### ~~问题 A4：`import random` 在推理路径中不应使用~~ ✅ 已修复 (2026-04-04)
**文件**：`backend/services/customer_service.py`

已修复（H4）：Agent 不可用时，若 `ENABLE_MOCK_DATA=false` 则返回 503 错误；若 `ENABLE_MOCK_DATA=true` 则返回 `degraded()` 响应明确标记为降级数据，不再用随机数伪装真实结果。

#### 问题 A5：Router 注册重复
**文件**：`backend/main.py:140,156`

```python
app.include_router(health.router, prefix="/api")        # 第一次
app.include_router(health.router, prefix="/admin")       # 第二次
```

`health.router` 被注册了两次（`/api/health` 和 `/admin/health`），这意味着同一个 router 实例被不同 prefix 共享。

**建议**：如果确实需要两个健康检查端点，显式创建两个 router 实例以避免潜在的标签和文档冲突。

#### 问题 A6：缺少 Alembic 数据库迁移
当前项目使用 `seed_all.sql` 手动建表，无版本化迁移。`pyproject.toml` 中声明了 `alembic>=1.13.0` 依赖但未使用。

**建议**：
```bash
alembic init alembic
# 配置 alembic/env.py 使用 settings.DATABASE_URL
alembic revision --autogenerate -m "initial"
alembic upgrade head
```

### 3.3 🟡 前端问题（中优先级）

#### 问题 F1：Pinia Store 缺少错误类型化
**文件**：`frontend/src/stores/useCustomerStore.js`

```javascript
catch (e) { this.error = e.message }
```

所有 Store 的 catch 块仅存储 `e.message` 字符串，丢失了错误类型（网络错误 vs 业务错误 vs 超时）。

**建议**：
```javascript
catch (e) {
  this.error = {
    message: e.message,
    code: e.response?.status,
    isNetwork: !e.response,
  }
}
```

#### 问题 F2：全量注册 Element Plus 图标
**文件**：`frontend/src/main.js:34-44`

注册了 40+ 个图标组件为全局组件，即使大部分页面只用到少数几个。

**建议**：按需导入，或使用 `unplugin-icons` + `unplugin-auto-import` 自动按需加载：
```javascript
// vite.config.js
import Icons from 'unplugin-icons/vite'
import IconsResolver from 'unplugin-icons/resolver'
import AutoImport from 'unplugin-auto-import/vite'
```

#### 问题 F3：ECharts 未按需引入所有需要的组件
**文件**：`frontend/src/main.js:17-18`

当前手动 `use()` 注册了部分图表和组件，但可能遗漏某些页面需要的组件（如 `ToolboxComponent`、`VisualMapComponent`）。

**建议**：使用 `echarts/core` 的按需导入是正确的，但建议在遇到 "xxx is not registered" 错误时，集中维护一个 `echarts-setup.js` 文件。

#### 问题 F4：大型 SFC 文件
- `OpenClaw.vue` — 19,566 bytes
- `FraudDetection.vue` — 16,416 bytes
- `CustomerAnalysis.vue` — 14,866 bytes

单个 Vue 文件超过 200 行 `<script>` 时，应考虑拆分为子组件 + composable。

**建议**：
```
views/business/CustomerAnalysis/
  ├── index.vue          # 主页面（组装层）
  ├── CustomerKPIs.vue   # KPI 卡片区
  ├── SegmentChart.vue   # 客群分布图
  ├── ChurnTable.vue     # 流失风险表
  └── useCustomerAnalysis.js  # 逻辑 composable
```

#### 问题 F5：Token 存储在 localStorage
**文件**：`frontend/src/api/request.js:13`、`frontend/src/stores/useAuthStore.js:7`

localStorage 容易受 XSS 攻击获取 Token。

**建议**（按安全级别递增）：
1. 短期：当前方案可接受，确保无 XSS 漏洞
2. 中期：使用 HttpOnly Cookie 存储 Token（需后端配合 Set-Cookie）
3. 长期：实现 CSRF Token + HttpOnly Cookie 双重保护

### 3.5 🟣 ML 模块

#### 问题 M1：ML 配置中模块级副作用
**文件**：`ml/config.py:29-32`

```python
for d in [MODELS, LOGS, CKPT_DIR, ...]:
```

```python
def ensure_dirs():
```
