# 数据库与状态存储决策（v4.0）

## 1. 结论

当前项目在 v4.0 第一阶段继续使用：

- **MySQL**：业务表、治理表、审计表、artifact metadata
- **Redis**：缓存、热点状态、队列辅助
- **PostgreSQL（LangGraph checkpoint）**：工作流断点恢复、HITL 恢复、thread 持久化
- **文件系统 / 对象存储占位**：artifact 内容与导出文件正文

业务主库不切换到 PostgreSQL；但 LangGraph checkpoint 使用 PostgreSQL 作为独立持久化后端。

---

## 2. 为什么继续使用 MySQL

- 当前 Docker、后端依赖、初始化脚本都已围绕 MySQL 组织
- 将业务主库整体切换到 PostgreSQL 会同时改动 ORM、部署、初始化、迁移脚本
- 当前目标是完成 v4.0 架构落地，而不是更换数据库体系
- MySQL 足以承载第一阶段的业务与治理元数据
- LangGraph checkpoint 独立使用 PostgreSQL，不改变业务主库与治理主库的既有结构

## 4. 关键表设计原则

### 4.1 `runs`

每一次外部请求或工作流执行都必须落一条 `runs` 记录，至少包含：

- `request_id`
- `thread_id`
- `entrypoint`
- `workflow_name`
- `workflow_version`
- `status`
- `started_at`
- `ended_at`
- `latency_ms`
- `total_tokens`
- `total_cost`

### 4.2 `run_steps`

用于回放与审计每个步骤，至少包含：

- `run_id`
- `parent_step_id`
- `step_type`
- `step_name`
- `agent_name`
- `tool_name`
- `model_name`
- `prompt_id`
- `prompt_version`
- `policy_version`
- `handoff_from`
- `handoff_to`
- `status`
- `input_summary`
- `output_summary`
- `guardrail_hits_json`
- `token_usage_json`
- `cost_amount`
- `retry_count`
- `artifact_ids_json`
- `error_message`
- `started_at`
- `ended_at`

### 4.3 `artifacts`

所有关键产物都以 artifact 形式登记，至少包含：

- `artifact_type`
- `artifact_uri`
- `content_type`
- `summary`
- `metadata_json`

数据库中只保存引用与摘要，不强制把大文件正文直接塞进表里。

### 4.4 `releases` / `release_items` / `release_rollbacks`

Release Center 必须具备正式治理表，而不是页面占位：

- `releases`：记录发布批次、发布对象、版本、发布人、发布时间、状态
- `release_items`：记录一次发布涉及的 prompt / policy / workflow / config 具体项
- `release_rollbacks`：记录回滚原因、回滚目标版本、执行人、结果摘要、关联审批

### 4.5 `memory_records` / `faq_documents` / `memory_feedback`

Knowledge & Memory Center 必须落地到正式表设计：

- `faq_documents`：FAQ 文档、知识分组、启停状态、来源说明
- `memory_records` 至少包含：
  - `customer_id`
  - `memory_kind`（`semantic` / `episodic`）
  - `source_type`
  - `source_run_id`
  - `source_message_id`
  - `content_summary`
  - `risk_level`
  - `pii_flag`
  - `expires_at`
  - `is_active`
  - `validated_by`
  - `feedback_score`
  - `created_at`
  - `updated_at`
- `memory_feedback`：记录人工反馈、自动反馈、禁用原因、过期原因等治理信号
  - `feedback_type`（enum）：`disable` / `expire` / `flag_pii` / `human_review` / `auto`
  - `reason`：操作备注（可选）
  - `operated_by`：操作人
  - `created_at`：操作时间

#### 记忆治理操作语义

| 操作 | 修改字段 | 语义 | 典型场景 | 可逆性 |
|------|---------|------|---------|--------|
| **强制过期**（expire） | `expires_at = NOW()` | 记忆时效到期，不再参与推荐/决策，但 `is_active` 仍为 1 | 促销偏好过时、季节性记忆失效 | 可逆（修改 `expires_at` 即可恢复） |
| **停用**（disable） | `is_active = 0` | 记忆被管理员主动关闭，完全退出使用 | 错误记忆、违规内容、PII 泄露 | 不可逆（前端无"重新启用"入口） |

- 两个操作**互相独立**：一条记忆可以同时处于"已过期 + 启用"或"未过期 + 停用"状态
- 两个操作都会自动写入 `memory_feedback`（分别为 `expire` / `disable` 类型）和 `audit_logs`
- Agent 读取记忆时应同时检查 `is_active = 1` 且 `expires_at IS NULL OR expires_at > NOW()`

### 4.6 `action_ledgers`

所有高风险写操作都必须记录 action ledger，至少包含：

- `action_type`
- `target_type`
- `target_id`
- `idempotency_key`
- `requested_by`
- `approved_by`
- `status`
- `result_summary`
- `created_at`
- `updated_at`

### 4.7 RBAC 基础表

后台身份认证与权限控制至少需要：

- `users`
- `roles`
- `permissions`
- `user_roles`
- `role_permissions`

### 4.8 Eval 治理表

Eval Center 除了数据集与实验，还必须具备：

- `evaluators`：记录 evaluator 定义、适用场景、评分规则、版本
- `eval_online_samples`：记录线上抽样样本、来源 run、导入批次、抽样说明

## 5. 状态与记忆

### 5.1 短期状态

- 工作流运行态保存在 LangGraph state 中
- Redis 可作为临时缓存和热点状态辅助层
- 不在 Redis 长期存放核心业务真相数据

### 5.2 会话恢复

- 使用 PostgreSQL（LangGraph checkpoint）保存 thread 断点
- 风控 HITL、客服多轮会话必须支持恢复
- 联调、演示与部署环境统一使用 PostgreSQL checkpoint

### 5.3 长期记忆

- 仅 OpenClaw 允许长期用户记忆
- `memory_records` 必须带有 `memory_kind` / `source` / `risk_level` / `pii_flag` / `expires_at` / `is_active`
- 禁止把内部风控阈值、员工备注、敏感原文写入用户记忆

## 6. Artifact 存储策略

### 6.1 当前阶段

第一阶段可采用：

- MySQL 记录 metadata
- 本地文件目录或挂载卷保存正文产物
- 导出文件使用统一命名路径管理

### 6.2 推荐路径规范

- `customer/{customer_id}/{date}/{artifact_id}`
- `forecast/{store_id}/{sku}/{date}/{artifact_id}`
- `risk/{case_id}`
- `reports/{run_id}`

必须至少对以下字段建立索引：

- `runs.request_id`
- `runs.thread_id`
- `runs.workflow_name`
- `run_steps.run_id`
- `run_steps.parent_step_id`
- `artifacts.run_id`
- `review_cases.status`
- `audit_logs.target_id`
- `memory_records.customer_id`
- `action_ledgers.idempotency_key`
- `releases.status`
- `user_roles.user_id`

### 7.2 审计要求

以下操作必须写入 `audit_logs`：

- Prompt 发布
- Policy 发布
- Review 审批动作
- Release 发布与回滚
- 高风险人工修改
- 生产环境配置变更

## 8. Git 协作要求

数据库变更必须遵循：

1. 先更新 ADR 与 `方案_11_v4开发任务清单.md`
2. 再更新 `backend/db/init.sql` 或迁移脚本
3. 明确给出回滚方式
4. PR 中说明是否影响已有数据

数据库相关分支推荐：

- `feature/db-governance-tables`
- `feature/db-artifact-store`
- `feature/db-review-audit`

---

## 9. 暂缓事项

当前阶段暂缓：

- 业务主库迁移到 PostgreSQL
- 向量数据库单独拆分
- 大规模对象存储治理平台
- 跨租户复杂分库分表

当前数据库方案的目标是：**以最小迁移成本支撑 v4.0 的工作流、治理与审计闭环。**
