## 一、五大平台架构对比

### 1.1 核心架构模型对比

| 维度 | Claude Code | LangGraph | CrewAI | AutoGen | Dify |
|------|------------|-----------|--------|---------|------|
| **架构范式** | 单 Agent Loop + 多 Tool | 状态图 (StateGraph) | 角色 Crew 协作 | 对话式多 Agent | 可视化工作流 + Agent Node |
| **编排方式** | 循环：LLM→Tool→Result→LLM | 有向图：节点+边+条件路由 | 任务链：Crew→Task→Agent | 对话轮转：Agent↔Agent | 画布拖拽：Node→Edge→Node |
| **状态管理** | Zustand Store + 消息历史 | Checkpoint (PG/Memory) | Crew State + Memory | 对话上下文 | 变量传递 + 会话变量 |
| **HITL 支持** | 权限提示 (ask mode) | `interrupt()` + `Command` | Task 回调 | UserProxy Agent | 人工审核节点 |
| **Tool 系统** | 41 内置 + MCP 协议 | LangChain Tools | CrewAI Tools | 函数工具 | 内置 + 插件 + MCP |
| **安全模型** | 5 层权限 + Bash AST 解析 | 无内置（需自建） | Task Guardrails | 代码沙箱 | 插件隔离 |
| **可观测性** | GrowthBook + Datadog | LangSmith | 内置日志 | 对话日志 | Langfuse/Arize/Opik |
| **适用场景** | CLI 代码助手 | 复杂工作流编排 | 快速多 Agent 原型 | 对话式协作 | 可视化 AI 应用构建 |

### 1.2 Agent 架构模式分类（来自 2025 Agent Architecture Guide）

| 模式 | 适用场景 | 本项目采用？ |
|------|---------|-------------|
| **Single Agent + Tools** | 聚焦型任务，8-10 工具以内 | ✅ OpenClawAgent (对话式) |
| **Sequential Agents** | 多阶段流水线 | ✅ business_overview workflow |
| **Agents Hierarchy + Parallel** | 复杂研究、多领域 | ✅ SupervisorAgent → 分析 Agents |
| **Single Agent + Router** | 意图路由分发 | ✅ SupervisorAgent 路由功能 |

**结论**：本项目 v4.0 的 「Supervisor + 4 专业 Agent + LangGraph workflow」架构模式，恰好是**Agents Hierarchy + HITL + Router**的组合，与业界最佳实践高度吻合。

## 二、从平台对比中提炼的新增改进点

基于本次对比分析，在 `05_代码架构分析与改进.md` 的 6 个改进基础上，新增 4 个改进点：

### 改进 7：Task Output Guardrails（借鉴 CrewAI）

**CrewAI 的核心思想**：每个 Task 可以绑定 guardrails 函数/LLM 检查输出质量。如果输出不满足条件，拒绝结果并要求 Agent 重试。

**方案**：为本项目的关键 workflow 节点添加输出验证：

```python
# backend/governance/guardrails/output_validator.py

class OutputValidator:
    """验证 Agent/Service 输出是否满足质量标准"""

    async def validate(self, output: dict, rules: list[ValidationRule]) -> ValidationResult:
        for rule in rules:
            if rule.type == 'schema':
                # 检查输出 schema 完整性
                if not self.check_schema(output, rule.expected_schema):
                    return ValidationResult(passed=False, reason=f"缺少必要字段: {rule.missing_fields}")

            elif rule.type == 'range':
                # 检查数值范围合理性
                if not self.check_range(output, rule.field, rule.min, rule.max):
                    return ValidationResult(passed=False, reason=f"{rule.field} 超出合理范围")

            elif rule.type == 'llm_judge':
                # 用 LLM 判断输出质量（类似 Claude Code 的 YOLO classifier）
                score = await self.llm_judge(output, rule.criteria)
                if score < rule.threshold:
                    return ValidationResult(passed=False, reason=f"LLM 质量评分 {score} < {rule.threshold}")

        return ValidationResult(passed=True)
```

**落地场景**：
- InsightComposerAgent 输出必须包含 `executive_summary`, `risk_highlights`, `action_plan`
- FraudScoringService 输出 `final_score` 必须在 0-100 范围
- 经营分析 workflow 的最终报告至少包含 3 个 artifact_refs

### 改进 8：三层上下文压缩（借鉴 Claude Code）

**Claude Code 的核心思想**：三层分级压缩——微压缩（无 API 调用，截断过长内容）→ 自动压缩（LLM 摘要）→ 记忆整合（后台异步）。

**方案**：

```text
Layer 1: 微压缩 (Micro Compact) — service 调用层
  → artifact 内容超过 2000 tokens 时，只传 summary + 前 500 tokens
  → 不需要额外 LLM 调用，零成本

Layer 2: 自动压缩 (Auto Compact) — workflow 层
  → OpenClaw 多轮对话超过 10 轮时，用 LLM 压缩前 8 轮为摘要
  → 触发条件：total_tokens > 80% context_window

Layer 3: 记忆整合 (Memory Consolidation) — 异步层
  → 每次 workflow 结束后，提取关键学习存入 memory_records
  → 下次同用户请求时注入历史记忆
```

### 改进 9：错误分类与智能重试（借鉴 Claude Code + Dify）

**Claude Code 的核心思想**：41KB 错误分类代码 + 28KB 重试逻辑。不是所有错误都重试：429 限流等待重试，401 认证直接报错，prompt-too-long 自动压缩后重试。

**Dify 的核心思想**：工作流错误分支 + 超时处理 + 重试配置。

**方案**：

```python
# backend/core/error_handler.py

class ErrorClassifier:
    """LLM API 错误分类器"""

    def classify(self, error: Exception) -> ErrorAction:
        if isinstance(error, RateLimitError):
            return ErrorAction(
                type='retry_with_backoff',
                wait_seconds=self.extract_retry_after(error),
                max_retries=3
            )
        elif isinstance(error, ContextTooLongError):
            return ErrorAction(
                type='compact_and_retry',
                strategy='auto_compact'
            )
        elif isinstance(error, AuthenticationError):
            return ErrorAction(
                type='abort',
                message='LLM API 认证失败，请检查 API Key'
            )
        elif isinstance(error, ModelUnavailableError):
            return ErrorAction(
                type='fallback_model',
                fallback='deepseek-chat'  # 切换到备用模型
            )
        else:
            return ErrorAction(
                type='retry',
                max_retries=2,
                backoff='exponential'
            )
```

### 改进 10：SSE 实时进度推送（借鉴 Claude Code Streaming-First + Dify）

**Claude Code 的核心思想**：流式优先，mid-stream 检测 tool_use 并即时反馈。
**Dify 的核心思想**：workflow 节点执行状态实时推送到前端。

**方案**：

```python
# backend/routers/api/workflow_sse.py

@router.get("/api/workflow/{run_id}/stream")
async def stream_workflow_progress(run_id: str):
    """SSE 端点，实时推送 workflow 执行进度"""
    async def event_generator():
        async for event in workflow_progress_channel(run_id):
            yield {
                "event": event.type,  # step_started | step_completed | step_failed | final
                "data": json.dumps({
                    "step_name": event.step_name,
                    "agent_name": event.agent_name,
                    "status": event.status,
                    "progress_pct": event.progress_pct,
                    "message": event.message,  # "正在执行客户分析..."
                    "partial_result": event.partial_result,
                })
            }
    return EventSourceResponse(event_generator())
```

前端对应：
```javascript
// frontend/src/composables/useWorkflowStream.js
const eventSource = new EventSource(`/api/workflow/${runId}/stream`)
eventSource.addEventListener('step_started', (e) => {
  const data = JSON.parse(e.data)
  updateStepStatus(data.step_name, 'running', data.message)
})
```

## 四、从各平台学到的「生产级」经验总结

### 4.1 Claude Code 生产级核心洞察

来自 Deep-Dive-Claude-Code 13 章分析的 5 个核心结论：

1. **循环不变** — 从 30 行教学版到 10MB+ 生产版，Agent 循环本质没变，变的是围绕它的工程层
2. **安全是骨架** — 300KB+ 安全代码（Bash AST 解析器 128KB + 安全分类 100KB + 权限引擎 96KB），不是锦上添花
3. **上下文是稀缺资源** — 三层压缩 + 微压缩 + 做梦记忆整合，都是为了在有限窗口内做无限的事
4. **可扩展性来自标准** — MCP 协议、插件系统、Skill 机制让能力边界无限扩展
5. **工程量是 10 倍** — 核心功能占 10% 代码，剩下 90% 是让它在生产环境可靠运行的工程支撑

### 4.2 LangGraph 生产级核心经验

1. **Checkpoint 是生命线** — 没有持久化，HITL 和崩溃恢复都不可能
2. **`interrupt()` + `Command` 是 HITL 的标准模式** — 暂停→等待人工→恢复执行
3. **State 显式传递** — 不依赖隐式全局状态，每个节点输入输出明确
4. **时间旅行调试** — Checkpoint 可以回退到任意步骤重新执行

### 4.3 CrewAI 生产级核心经验

1. **Start Small** — 先 2-3 Agent，别上来就 10 个
2. **定义清晰角色** — 避免 Agent 职责重叠
3. **早期加入 Memory** — 等后期加太晚，Agent 无法从历史中学习
4. **早期加入日志** — 多 Agent 系统排查困难，可观测性必须先行
5. **关注总成本** — LLM 调用 × Agent 数量 × 轮次数 = 实际成本

### 4.4 Dify 生产级核心经验

1. **Trace 是可持续迭代的最佳预测器** — 尤其是使用多 Tool 和知识源的 Agent
2. **Hybrid Retrieval** — 向量 + BM25 关键词混合检索比纯向量效果更好
3. **错误分支是必须的** — 每个关键节点都要有失败分支，不能让一个节点崩溃整个 workflow
4. **插件隔离** — 外部工具在独立进程/沙箱中运行

## 五、本项目架构验证与定位

通过对比五大平台，本项目 v4.0 架构的定位更加清晰：

```text
本项目 ≈ LangGraph (工作流引擎)
       + Claude Code 式安全模型 (分层 Guardrails)
       + CrewAI 式角色分工 (5 个专业 Agent)
       + Dify 式可观测性 (Trace + Audit + 治理后台)
       + 自有业务 Service 层 (8+ 业务确定性服务)
```

### 架构优势确认

| 维度 | 本项目选择 | 理由 |
|------|-----------|------|
| **工作流引擎** | LangGraph | 有向图 + Checkpoint + HITL 原生支持，适合复杂业务流程 |
| **Agent 数量** | 5 个 | 与 Claude Code「少 Agent 多 Tool」+ CrewAI「Start Small」一致 |
| **Service 层** | 确定性服务（非 Agent） | 与 CrewAI「不需智能的步骤就用代码」一致 |
| **治理后台** | 独立控制台 | Dify 也有独立管理后台，是可观测性的入口 |
| **数据库选型** | MySQL(业务) + PG(checkpoint) | LangGraph 推荐 PG checkpoint，业务用关系型 |

### 需要补强的方面

| 维度 | 当前差距 | 参照平台 | 改进编号 |
|------|---------|---------|---------|
| 运行时安全检查 | Policy 表有但不执行 | Claude Code | 改进 3 |
| 错误韧性 | 无分类重试 | Claude Code + Dify | 改进 9 |
| 用户体感 | 无流式进度 | Claude Code + Dify | 改进 10 |
| 输出质量保障 | 无 output guardrails | CrewAI | 改进 7 |
| 上下文效率 | 无压缩策略 | Claude Code | 改进 8 |
