# 依赖、框架与外部服务决策（v4.0）

## 1. 结论

v4.0 继续基于当前项目已有依赖栈演进，在此基础上补充 Agent 编排、可观测性和治理能力相关依赖。

核心原则：**优先复用现有依赖，按需新增，不为“先进感”做无必要换栈。**

---

## 2. 当前保留的核心依赖

### 2.1 后端基础

- `fastapi`
- `uvicorn`
- `pydantic`
- `pydantic-settings`
- `sqlalchemy`
- `pymysql`
- `alembic`
- `redis[asyncio]`

### 2.2 业务与模型推理

- `pandas`
- `numpy`
- `scikit-learn`
- `xgboost`
- `lightgbm`
- `torch`
- `transformers`
- `jieba`
- `gensim`
- `statsmodels`
- `pmdarima`
- `lifetimes`
- `shap`
- `mlxtend`

以上依赖继续服务于 `ml/` 资产和 `backend/services/` 推理层，不因 v4.0 被废弃。

---

## 3. v4.0 新增依赖

### 3.1 Agent 与工作流

建议新增：

- `langgraph`
- `langchain-core`
- `langchain-openai`
- `langgraph-checkpoint-postgres`

用途：

- 工作流编排
- Agent 节点组织
- LLM 接入
- 断点恢复

联调、演示与部署环境统一使用 PostgreSQL checkpoint。

### 3.2 可观测性

建议新增：

- `langsmith`（或通过环境变量接入）

用途：

- LLM / workflow trace 追踪
- 调试路由错误
- 分析 token、耗时、失败链路

### 3.3 长期记忆（可选）

建议新增：

- `mem0ai`（仅 OpenClaw 场景可选）

用途：

- 用户偏好与客服多轮长期记忆

---

## 4. 模型与外部服务策略

### 4.1 LLM 提供方

当前架构优先支持兼容 OpenAI SDK 的提供方：

- DeepSeek
- Qwen 兼容模式
- OpenAI 官方接口

### 4.2 选择原则

- 中文能力优先
- 接口稳定优先
- 价格可控优先
- 能被统一配置切换优先

### 4.3 配置方式

统一使用环境变量：

- `LLM_API_KEY`
- `LLM_BASE_URL`
- `LLM_MODEL_NAME`
- `LANGCHAIN_TRACING_V2`
- `LANGCHAIN_API_KEY`
- `LANGCHAIN_PROJECT`
- `MEM0_API_KEY`

---

## 5. 当前明确不引入的依赖方向

为了适配现有仓库和 Git 节奏，当前阶段明确不做以下依赖升级：

- 不引入 Next.js / React 全栈体系
- 不将 PostgreSQL 作为业务主库迁移的前置条件；仅将其用于 LangGraph checkpoint
- 不引入重量级消息队列作为第一阶段硬依赖
- 不引入 Kubernetes 作为本地开发前置要求
- 不做完整微服务治理平台化改造

---

## 6. Docker 与部署约束

### 6.1 当前部署形态

继续使用：

- `docker-compose.yml` 用于本地开发
- `docker-compose.prod.yml` 用于生产部署
- `nginx` 用于前后端代理与静态资源服务

### 6.2 依赖引入原则

新增依赖必须满足：

- 能在当前 `backend/pyproject.toml` 中安装
- 不要求额外的复杂系统级服务才能本地运行
- 有明确使用场景与替代关系
- 能写入 `方案_11_v4开发任务清单.md` 形成独立开发任务

---

## 7. Git 协作要求

依赖升级必须单独成任务，禁止把依赖升级和业务重构揉在一个 PR 里。

推荐分支：

- `feature/deps-langgraph`
- `feature/deps-langsmith`
- `feature/deps-openclaw-memory`

提交说明必须写清楚：

- 新增了什么依赖
- 用于哪个工作流或 Agent
- 是否影响 Docker 构建
- 是否需要更新 `.env.example`

---

## 8. 最终原则

v4.0 的依赖策略不是“追最新框架”，而是：

- 保住已有模型与后端资产
- 补足工作流编排和治理能力
- 让开发任务能被拆成原子化 PR
- 让现有团队能在当前仓库里稳定推进

这套依赖决策服务于工程落地，而不是概念堆叠。
