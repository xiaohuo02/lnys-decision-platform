# Claude Code 架构深度分析与本项目改进方案

> **分析日期**: 2026-04-02
> **分析来源**: Claude Code 源码逆向工程（ComeOnOliver/claude-code-analysis）、官方文档、社区解析文章
> **目的**: 从 Claude Code 的工业级 Agent 架构中提炼可迁移到本项目（柠优生活大数据智能决策平台 v4.0）的关键设计模式

## 一、Claude Code 架构全景

### 1.1 核心架构模型

Claude Code 是一个**单进程 Agent Runtime + 终端 UI**。其核心是一个 **Agentic Loop**：

```text
用户输入
  → QueryEngine (对话循环编排器)
  → LLM API 调用 (SSE 流式)
  → 解析 tool_use blocks
  → 逐个 Tool 权限检查 → 执行 → 收集结果
  → 结果注入消息历史
  → LLM 判断是否需要更多 tool 调用
  → 循环 or 输出最终文本响应
```

**关键设计点**：
- **流式优先 (Streaming-First)**：API 响应使用 SSE，mid-stream 就检测到 tool_use 并触发执行
- **一次 API 调用可返回多个 tool_use blocks**，按序执行
- **循环终止条件**：LLM 输出纯文本响应（无 tool_use）

### 1.2 六大子系统

| 子系统 | 职责 | 规模 |
|--------|------|------|
| **QueryEngine** | 对话循环编排、消息管理、重试、用量追踪 | ~46KB 核心 |
| **Tool System** | 41个工具，统一接口：Schema → 权限 → 执行 → 结果 | 184 文件 |
| **Permission System** | 5层权限检查、沙箱隔离、Hook 扩展 | 多层嵌套 |
| **State Management** | Zustand 风格 Store + React Context | 6 文件 |
| **Context & Memory** | CLAUDE.md 发现、上下文压缩、会话记忆 | 多服务 |
| **Task/Agent System** | 子 Agent 生成、后台任务、团队协调 | 12+ 文件 |

### 1.3 技术栈

- **TypeScript + Bun**（打包与特性门控）
- **React + Ink**（终端 UI 渲染）
- **Zod v4**（所有 Schema 验证）
- **Commander.js**（CLI 解析）
- **Zustand 风格** Store（状态管理）

## 二、7 个关键架构模式深度解析

### 模式 1：统一工具接口 (Uniform Tool Interface)

Claude Code 的每个工具都实现同一个接口：

```typescript
interface Tool {
  name: string
  description: string
  inputSchema: JSONSchema         // Zod 验证的输入 Schema
  isEnabled(): boolean            // 是否在当前环境可用
  checkPermissions(input): PermissionResult  // allow | deny | ask
  execute(input, context): ToolResult        // 实际执行逻辑
  renderUI(progress): ReactNode             // 终端 UI 渲染
}
```

**核心原则**：与外部世界的每次交互都通过已注册的 Tool。
- 声明式 JSON Schema 输入
- 执行前的权限检查
- 执行期间的进度报告
- 结构化结果输出

### 模式 2：五层权限安全模型 (Layered Permission Model)

每次 Tool 调用经过 **5 层检查**：

```text
Layer 1: Tool 自身的 checkPermissions()
  ↓ (Bash 检查破坏性命令, Write 检查文件路径)
Layer 2: Settings allowlist / denylist
  ↓ (glob 模式匹配 Bash(npm:*), Read(~/project/**))
Layer 3: Sandbox policy
  ↓ (路径、命令、网络访问限制)
Layer 4: Permission mode
  ↓ (default | acceptEdits | plan | auto | bypassPermissions)
Layer 5: Hook overrides
  ↓ (PreToolUse hooks 可 approve/block/modify)
```

**5 种权限模式**：
- `default`：所有操作都询问
- `acceptEdits`：文件编辑自动批准，Shell 命令仍询问
- `plan`：只读模式，只能查看不能修改
- `auto`：自动评估 + 背景安全检查
- `bypassPermissions`：全部自动批准

### 模式 3：上下文工程 (Context Engineering)

Claude Code 的上下文注入是**分层管道**：

```text
System Context (OS, Shell, Platform, Git status)
  + User Context (CLAUDE.md 项目级/用户级/全局级)
  + Memory Files (~/.claude/memory/ 跨会话学习)
  + Session History (可恢复/分叉的会话)
  → 合并为 System Prompt
```

**上下文压缩策略**：
- **自动压缩 (autoCompact)**：接近上下文窗口限制时触发
- **微压缩 (microCompact)**：选择性修剪
- **会话记忆持久化 (sessionMemoryCompact)**：关键信息跨压缩保留
- **Pre/Post Hook**：用户可控制哪些信息不被压缩

### 模式 4：任务与子 Agent 系统 (Task & SubAgent)

```text
7 种任务类型:
  local_bash       → 本地 Shell
  local_agent      → 子 Agent (AgentTool 生成)
  remote_agent     → 远程 Agent
  in_process_teammate → 进程内队友
  local_workflow   → 多步骤工作流
  monitor_mcp      → MCP 服务器监控
  dream            → 自动后台任务

生命周期:
  pending → running → completed | failed | killed

关键设计:
  - 每个子 Agent 有独立上下文窗口
  - 任务输出持久化到磁盘
  - AbortController 支持取消
  - 完成后返回摘要给父 Agent
```

### 模式 5：Hook 驱动的可扩展性 (Hook-Driven Extensibility)

```text
几乎所有关键动作都有 Pre/Post Hook:
  PreToolUse   → 工具调用前（可拦截/修改）
  PostToolUse  → 工具调用后（可检查结果）
  PreCompact   → 上下文压缩前
  PostCompact  → 上下文压缩后
  PromptHook   → 提示词注入

Hook 能力:
  - Shell 脚本形式执行
  - 可 approve / block / modify 工具调用
  - 可注入额外上下文
  - 可在文件编辑后运行 linter/formatter
```

### 模式 6：配置层级合并 (Configuration Hierarchy)

```text
Defaults
  → ~/.claude/settings.json (用户全局)
  → .claude/settings.json (项目级, checked into VCS)
  → .claude/settings.local.json (项目本地, gitignored)
  → CLI flags
  → Environment variables

后者覆盖前者。团队级配置入 VCS，个人偏好本地覆盖。
```

### 模式 7：成本与可观测性追踪 (Cost & Observability)

```text
每次对话追踪:
  - Token 计数 (input / output / cache_read / cache_write)
  - 美元成本 (按模型定价表)
  - 会话持续时间
  - 网络搜索请求数
  - 文件变更指标
  - 每个 Tool 的调用次数与延迟
```

## 四、可落地的改进方案

### 改进 1：统一 Service/Tool 调用协议 (Uniform Service Protocol)

**从 Claude Code 学到的核心思想**：每次与外部世界的交互都通过统一接口。

**方案**：为 `backend/services/*` 定义一个统一的调用协议：

```python
# backend/core/service_protocol.py

class ServiceCallContext:
    """每次 service 调用的上下文"""
    run_id: str
    step_id: str
    caller: str          # agent_name or workflow_name
    permission_mode: str  # auto | check | hitl
    trace_enabled: bool

class ServiceResult:
    """统一 service 返回结构"""
    success: bool
    data: dict | None
    artifact_ref: str | None
    error: str | None
    fallback_used: bool
    metrics: ServiceMetrics  # latency_ms, tokens, cost

class ServiceMetrics:
    latency_ms: int
    tokens_used: int
    cost_amount: float

def service_call(
    service_name: str,
    method: str,
    input_data: dict,
    context: ServiceCallContext,
) -> ServiceResult:
    """
    统一 service 调用入口:
    1. 输入 Schema 校验
    2. 权限检查 (如果需要)
    3. 执行
    4. 自动写入 run_steps
    5. 返回统一结果
    """
```

**落地价值**：
- 所有 service 调用自动产生 trace 记录
- 权限检查可统一拦截
- 降级与错误处理标准化
- 为 Ops Copilot 提供结构化诊断数据

### 改进 2：运行时 Trace 自动埋点管道 (Auto Trace Pipeline)

**从 Claude Code 学到的核心思想**：每个 Tool 调用都被追踪，包括输入摘要、输出摘要、耗时、token、成本。

**方案**：实现 `TraceMiddleware` 自动装饰 workflow 节点：

```python
# backend/governance/trace_center/trace_middleware.py

class TraceMiddleware:
    """装饰 LangGraph 节点，自动记录 run_step"""

    async def wrap_node(self, node_fn, node_name, run_id):
        step_id = uuid4()
        started_at = datetime.now()

        # 记录开始
        await self.record_step_start(run_id, step_id, node_name)

        try:
            result = await node_fn()
            # 记录成功
            await self.record_step_end(
                step_id, status='completed',
                output_summary=self.summarize(result),
                latency_ms=(datetime.now() - started_at).ms
            )
            return result
        except Exception as e:
            # 记录失败
            await self.record_step_end(
                step_id, status='failed',
                error_message=str(e)
            )
            raise
```

### 改进 3：分层 Policy 执行引擎 (Layered Policy Engine)

**从 Claude Code 学到的核心思想**：5 层权限检查 — 工具自检 → allowlist/denylist → sandbox → mode → hook。

**方案**：为本项目实现 3 层（适配当前规模）：

```text
Layer 1: Service 自检 (service-level check)
  → 例如 FraudScoringService 检查 final_score >= threshold 时标记 hitl_required

Layer 2: Policy Rules 匹配 (policy_center)
  → 从 policies 表读取 active 规则
  → 匹配 input_guard / output_guard / tool_guard
  → 命中则 block 或 flag

Layer 3: HITL 中断 (hitl_center)
  → 高风险结果触发 LangGraph interrupt()
  → 创建 review_case
  → 等待人工 approve / edit / reject
```

### 改进 4：上下文压缩策略 (Context Compaction)

**从 Claude Code 学到的核心思想**：自动压缩旧消息，保留关键上下文。

**方案**：为 InsightComposerAgent 和 OpenClawAgent 实现 artifact 摘要注入：

```text
不直接传完整 artifact 内容给 LLM
→ 传 artifact_ref + summary (< 500 tokens)
→ 只有 Agent 明确需要详情时才读取完整内容
→ 多轮对话时自动压缩历史消息为摘要

实现路径:
1. artifact 存储时必须生成 summary 字段
2. Agent 收到的是 artifact_refs + summaries
3. Agent 可调用 read_artifact_detail(ref) 获取完整内容
4. OpenClaw 多轮对话超过阈值时触发 compact
```

### 改进 5：PreToolUse / PostToolUse Hook 管道

**从 Claude Code 学到的核心思想**：几乎每个动作都有 pre/post hook 点。

**方案**：在 workflow 的 service 调用前后插入 hook：

```python
# backend/governance/hooks.py

class HookPipeline:
    """Pre/Post hook 管道"""

    async def pre_service_call(self, service_name, input_data, context):
        """在 service 调用前执行:
        - 输入校验 (input_guard policy)
        - PII 脱敏
        - 审计日志预记录
        - 可 block 调用
        """
        for hook in self.get_hooks('pre_service_call', service_name):
            result = await hook.execute(input_data, context)
            if result.action == 'block':
                return HookResult(blocked=True, reason=result.reason)
        return HookResult(blocked=False)

    async def post_service_call(self, service_name, result, context):
        """在 service 调用后执行:
        - 输出校验 (output_guard policy)
        - 敏感信息检查
        - artifact 自动保存
        - 高风险标记
        """
```

### 改进 6：成本追踪器 (Cost Tracker)

**从 Claude Code 学到的核心思想**：实时追踪每次对话的 token 和成本。

**方案**：

```python
# backend/governance/trace_center/cost_tracker.py

class CostTracker:
    """会话级成本追踪"""

    def __init__(self, run_id: str):
        self.run_id = run_id
        self.total_tokens = 0
        self.total_cost = 0.0
        self.step_costs = []

    def record_llm_call(self, model, input_tokens, output_tokens):
        cost = self.calculate_cost(model, input_tokens, output_tokens)
        self.total_tokens += input_tokens + output_tokens
        self.total_cost += cost

    async def flush_to_db(self):
        """将累积的成本写入 runs 表"""
        update runs set total_tokens=self.total_tokens,
                        total_cost=self.total_cost
        where run_id=self.run_id
```

## 六、Claude Code 给本项目的"反直觉"启示

### 6.1 Agent 越少越好

Claude Code 本身就是**一个 Agent + 大量 Tool**。它不是把每个能力做成独立 Agent，而是把能力做成 Tool，由单一 Agent Loop 编排。

**对本项目的启示**：v4.0 已经采用了「少量 Agent + 大量 Service」的正确方向。5 个 Agent 的设计是合理的，不要再增加 Agent 数量，而应该增加 Service 和 Tool 的丰富度。

### 6.2 权限检查在运行时，不只在设计时

Claude Code 的 5 层权限不是文档上的概念，而是**每次 tool 调用都实际执行**的代码路径。

**对本项目的启示**：policies 表和 guardrails 设计已有，但**运行时执行**是缺失的。需要补上 Policy 执行引擎，让每次 service 调用都经过实际的规则匹配。

### 6.3 上下文工程 > Prompt 工程

Claude Code 花了大量架构成本在「如何把正确的上下文在正确的时机注入给 LLM」上——CLAUDE.md 发现、Memory 文件、Git 状态、上下文压缩。

**对本项目的启示**：InsightComposerAgent 和 SupervisorAgent 的效果，很大程度取决于它们收到的 artifact 摘要质量。应该投资在 artifact 的 summary 生成质量上，而不是反复调 prompt。

### 6.4 Streaming-First 提升体感

Claude Code 的流式优先设计让用户感觉很快——即使实际处理需要 30 秒，用户也能看到中间过程。

**对本项目的启示**：经营总览分析 Workflow 可能需要 30+ 秒。应该在前端实现 SSE 推送，让用户实时看到「正在执行客户分析...」「正在执行销售预测...」等中间状态，而不是显示一个 loading 等 30 秒。

### 6.5 Hook 是扩展性的核心

Claude Code 把 hook 做成一等公民——几乎每个动作都有 pre/post hook。这使得用户可以在不修改核心代码的情况下定制行为。

**对本项目的启示**：governance 层的 hook 管道（PreToolUse / PostToolUse）不是 P4 的「锦上添花」，而应该是 P3 的核心组成部分。有了 hook 管道，Policy Center、Audit Center、未来的自定义插件都可以以非侵入式的方式接入。
